## Chapter Thesis

동시성(concurrency)은 **여러 작업을 겹쳐서 진행하는 설계**이고, 병렬성(parallelism)은 **여러 CPU 코어에서 실제로 동시에 실행하는 실행 방식**입니다. Python을 깊이 이해하려면 `threading`, `asyncio`, `subprocess`, `concurrent.futures`가 같은 문제를 해결하는 네 가지 이름이 아니라, 서로 다른 비용 모델과 실패 모델을 가진 도구라는 점을 구분해야 합니다.

Python 3.14에서는 `concurrent.interpreters`와 `InterpreterPoolExecutor`가 표준 라이브러리에 들어오면서, 프로세스보다 가벼운 격리 기반 병렬성 선택지가 생겼습니다. 따라서 이 챕터의 핵심 질문은 “스레드를 쓸까, 코루틴을 쓸까?”가 아니라 **작업의 병목, 상태 공유, 취소 방식, 배포 환경을 보고 가장 단순한 모델을 고르는 법**입니다.

## Item 67. Use subprocess to Manage Child Processes

### Central Question

Python 코드 안에서 외부 명령을 실행해야 할 때 무엇을 조심해야 할까?

### My Explanation

`subprocess`는 Python 프로세스 바깥의 프로그램을 실행하고, 표준 입력/출력/오류와 종료 코드를 다루는 표준 도구입니다.

```python
from subprocess import run

result = run(
    ["python", "--version"],
    capture_output=True,
    text=True,
    check=True,
)
print(result.stdout.strip())
```

좋은 사용 습관은 세 가지입니다.

1. shell 문자열보다 **인자 리스트**를 우선한다.
2. 실패를 숨기지 않도록 `check=True`를 고려한다.
3. 무한 대기를 막기 위해 긴 작업에는 `timeout`을 둔다.

```python
from subprocess import TimeoutExpired, run

try:
    run(["python", "slow_script.py"], timeout=3, check=True)
except TimeoutExpired:
    print("child process timed out")
```

### Takeaway

`subprocess`는 Python이 모든 일을 직접 하겠다는 도구가 아니라, 외부 도구와 안전하게 경계를 긋는 방법입니다. 입력을 분리하고, 종료 코드를 확인하고, 시간 제한을 두면 운영 가능한 코드가 됩니다.

## Item 68. Use Threads for Blocking I/O; Avoid for Parallelism

### Central Question

스레드는 언제 빠르고, 언제 기대만큼 빠르지 않을까?

### My Explanation

CPython의 일반 실행에서는 GIL 때문에 CPU-bound Python 바이트코드를 여러 스레드가 동시에 병렬로 실행하기 어렵습니다. 하지만 파일, 네트워크, 데이터베이스처럼 **대기 시간이 긴 blocking I/O**는 스레드로 겹쳐 실행하면 전체 시간이 줄어들 수 있습니다.

```python
from threading import Thread
from urllib.request import urlopen

urls = ["https://example.com", "https://www.python.org"]


def fetch(url: str) -> None:
    with urlopen(url, timeout=5) as response:
        print(url, response.status)

threads = [Thread(target=fetch, args=(url,)) for url in urls]
for thread in threads:
    thread.start()
for thread in threads:
    thread.join()
```

반대로 큰 숫자 계산, 이미지 처리, 압축처럼 Python 레벨 CPU 계산이 병목이라면 스레드만 늘려서는 병렬 속도 향상을 기대하기 어렵습니다. 이때는 프로세스, 네이티브 확장, 다중 인터프리터 같은 다른 모델을 검토해야 합니다.

### Takeaway

스레드는 “CPU를 더 쓰는 도구”라기보다 “기다리는 동안 다른 일을 하는 도구”에 가깝습니다. I/O-bound와 CPU-bound를 먼저 구분해야 합니다.

## Item 69. Use Lock to Prevent Data Races in Threads

### Central Question

GIL이 있는데도 왜 `Lock`이 필요할까?

### My Explanation

GIL은 메모리 안전을 돕지만, 사용자 수준의 논리적 불변식을 자동으로 지켜주지 않습니다. 여러 단계로 이루어진 읽기-수정-쓰기 작업은 스레드 전환 사이에서 깨질 수 있습니다.

```python
from threading import Lock, Thread

class Counter:
    def __init__(self) -> None:
        self.value = 0
        self._lock = Lock()

    def increment(self) -> None:
        with self._lock:
            self.value += 1

counter = Counter()
threads = [Thread(target=lambda: [counter.increment() for _ in range(1000)]) for _ in range(4)]
for thread in threads:
    thread.start()
for thread in threads:
    thread.join()

assert counter.value == 4000
```

하지만 락은 만능이 아닙니다. 락 범위가 크면 병렬성이 줄고, 락 순서가 꼬이면 deadlock이 생깁니다. 더 좋은 구조는 애초에 공유 mutable state를 줄이고, 메시지 전달이나 불변 데이터로 설계를 바꾸는 것입니다.

### Takeaway

공유 상태가 있으면 보호해야 합니다. 그러나 더 높은 수준의 목표는 “락을 잘 쓰는 코드”보다 “락이 덜 필요한 구조”입니다.

## Item 70. Use Queue to Coordinate Work Between Threads

### Central Question

스레드 사이에서 일을 어떻게 안전하게 전달할까?

### My Explanation

`queue.Queue`는 producer-consumer 구조를 만들 때 가장 기본이 되는 도구입니다. 내부적으로 필요한 동기화를 제공하므로 직접 리스트와 락을 조합하는 것보다 안전합니다.

```python
from queue import Queue
from threading import Thread

STOP = object()
work: Queue[int | object] = Queue()


def worker() -> None:
    while True:
        item = work.get()
        try:
            if item is STOP:
                return
            print(item * item)
        finally:
            work.task_done()

thread = Thread(target=worker)
thread.start()

for number in [1, 2, 3]:
    work.put(number)
work.put(STOP)
work.join()
thread.join()
```

중요한 점은 종료 신호입니다. 작업 큐는 일을 전달할 뿐 아니라 “더 이상 일이 없다”는 상태도 명시해야 합니다. `STOP` sentinel, timeout, cancellation flag 등을 설계에 포함해야 합니다.

### Takeaway

큐는 단순한 자료구조가 아니라 스레드 사이의 계약입니다. 작업 단위, 종료 신호, 오류 전달 방식을 함께 설계해야 합니다.

## Item 71. Know How to Recognize When Concurrency Is Necessary

### Central Question

동시성을 도입하기 전에 무엇을 확인해야 할까?

### My Explanation

동시성은 성능을 올릴 수 있지만, 코드의 디버깅 난이도도 올립니다. 따라서 “느릴 것 같다”가 아니라 **측정된 병목**이 있어야 합니다.

동시성이 필요한 신호는 다음과 같습니다.

| 신호 | 해석 | 후보 모델 |
|---|---|---|
| 네트워크 응답 대기 시간이 대부분이다 | I/O-bound | `asyncio`, thread pool |
| 파일/DB 작업을 기다린다 | I/O-bound | thread pool, async driver |
| Python 계산이 CPU를 계속 점유한다 | CPU-bound | process pool, interpreter pool, native code |
| 외부 명령 실행이 많다 | process orchestration | `subprocess`, process pool |
| 수천 개 연결을 유지한다 | high concurrency I/O | `asyncio` |

가장 좋은 루프는 다음과 같습니다.

```text
측정 → 병목 분류 → 가장 단순한 동시성 모델 선택 → 실패/취소 처리 → 다시 측정
```

### Takeaway

동시성은 설계 결정입니다. 증거 없이 도입하면 성능보다 복잡도가 먼저 증가합니다.

## Item 72. Avoid Creating New Thread Instances for On-demand Fan-out

### Central Question

요청이 들어올 때마다 새 스레드를 만들면 왜 위험할까?

### My Explanation

스레드는 가볍지만 공짜는 아닙니다. 요청마다 무제한으로 스레드를 만들면 메모리, context switching, 외부 서비스 부하가 폭발할 수 있습니다.

```python
# 위험한 형태: 입력 크기만큼 스레드가 무제한 생성됨
# for job in jobs:
#     Thread(target=handle, args=(job,)).start()
```

대신 작업량을 제한하는 구조가 필요합니다.

```python
from concurrent.futures import ThreadPoolExecutor


def handle(job: int) -> int:
    return job * 2

with ThreadPoolExecutor(max_workers=8) as executor:
    results = list(executor.map(handle, range(100)))
```

`max_workers`는 단순한 튜닝값이 아니라 시스템을 보호하는 backpressure 장치입니다.

### Takeaway

동시성의 핵심은 많이 만드는 것이 아니라 제한하는 것입니다. fan-out에는 항상 상한이 있어야 합니다.

## Item 73. Understand How Using Queue for Concurrency Requires Refactoring

### Central Question

순차 코드를 큐 기반 동시성으로 바꾸면 무엇이 달라질까?

### My Explanation

큐를 도입한다는 것은 함수를 병렬로 감싸는 정도가 아닙니다. 코드를 **단계(stage)** 로 나누고, 단계 사이의 데이터 계약을 명확히 하는 일입니다.

```text
load input → parse → transform → save output
```

이 흐름을 큐로 바꾸면 각 단계는 독립 worker가 됩니다.

```python
from queue import Queue

raw_items: Queue[str] = Queue()
parsed_items: Queue[dict[str, str]] = Queue()
```

이때 새로 결정해야 할 것이 많습니다.

- 각 queue의 최대 크기
- 실패한 item의 재시도 여부
- worker 종료 순서
- 결과 순서 보존 필요 여부
- 예외를 어느 단계에서 수집할지

### Takeaway

큐 기반 설계는 처리량을 높일 수 있지만, 동시에 데이터 흐름을 명시적으로 재설계해야 합니다. “스레드 추가”가 아니라 “파이프라인 설계”입니다.

## Item 74. Consider ThreadPoolExecutor When Threads Are Necessary for Concurrency

### Central Question

직접 `Thread`를 관리하는 것보다 `ThreadPoolExecutor`가 좋은 이유는 무엇일까?

### My Explanation

`ThreadPoolExecutor`는 작업 제출, worker 수 제한, 결과 수집, 예외 전파를 같은 인터페이스로 제공합니다.

```python
from concurrent.futures import ThreadPoolExecutor, as_completed


def fetch_size(url: str) -> int:
    from urllib.request import urlopen
    with urlopen(url, timeout=5) as response:
        return len(response.read())

urls = ["https://example.com", "https://www.python.org"]

with ThreadPoolExecutor(max_workers=4) as executor:
    futures = [executor.submit(fetch_size, url) for url in urls]
    for future in as_completed(futures):
        print(future.result())
```

`future.result()`는 worker 내부 예외를 호출자 쪽에서 다시 발생시킵니다. 따라서 조용히 실패하는 background thread보다 운영/테스트가 쉽습니다.

### Takeaway

스레드가 필요하다면 먼저 executor를 고려합니다. 직접 thread lifecycle을 관리하는 코드는 마지막 선택지에 가깝습니다.

## Item 75. Achieve Highly Concurrent I/O with Coroutines

### Central Question

`asyncio`는 스레드와 무엇이 다를까?

### My Explanation

`asyncio`는 하나의 이벤트 루프 안에서 coroutine들이 `await` 지점에서 자발적으로 제어권을 넘기는 모델입니다. 수많은 I/O 작업을 관리할 때 스레드보다 메모리와 전환 비용이 낮을 수 있습니다.

```python
import asyncio

async def fetch_metadata(name: str) -> str:
    await asyncio.sleep(0.1)  # 네트워크 I/O 대기라고 가정
    return f"metadata:{name}"

async def main() -> None:
    results = await asyncio.gather(
        fetch_metadata("a"),
        fetch_metadata("b"),
        fetch_metadata("c"),
    )
    print(results)

asyncio.run(main())
```

핵심은 `await`가 없는 긴 CPU 작업이나 blocking I/O가 이벤트 루프를 멈춘다는 점입니다. async 함수 안에 들어갔다고 자동으로 비동기가 되는 것이 아닙니다.

### Takeaway

`asyncio`는 고동시성 I/O에 강합니다. 하지만 이벤트 루프를 막는 코드를 넣으면 전체 시스템의 반응성이 떨어집니다.

## Item 76. Know How to Port Threaded I/O to asyncio

### Central Question

기존 blocking I/O 코드를 async 구조로 옮길 때 무엇부터 바꿔야 할까?

### My Explanation

가장 먼저 경계를 나눠야 합니다.

1. 이미 async API가 있는 라이브러리는 `await` 기반으로 교체한다.
2. 아직 blocking API만 있는 코드는 thread executor로 감싼다.
3. 호출자까지 async 전파가 필요한지 확인한다.

```python
import asyncio
from pathlib import Path


def read_text_blocking(path: Path) -> str:
    return path.read_text(encoding="utf-8")

async def read_text_async(path: Path) -> str:
    return await asyncio.to_thread(read_text_blocking, path)
```

`asyncio.to_thread`는 blocking 함수를 별도 스레드에서 실행해 이벤트 루프를 막지 않게 해줍니다. 단, 이것은 임시 다리이지 모든 blocking 코드를 async로 만든다는 뜻은 아닙니다.

### Takeaway

async 전환은 호출 경계를 바꾸는 작업입니다. blocking 코드를 발견하고, async-native API와 thread bridge를 구분해야 합니다.

## Item 77. Mix Threads and Coroutines to Ease the Transition to asyncio

### Central Question

스레드 기반 시스템을 한 번에 전부 async로 바꿔야 할까?

### My Explanation

대부분의 현실적인 시스템은 한 번에 바꾸기 어렵습니다. 이때는 async event loop를 중심에 두고, 기존 blocking 함수는 제한된 thread pool로 이동시키는 방식이 안전합니다.

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor


def legacy_call(x: int) -> int:
    return x * x

async def main() -> None:
    loop = asyncio.get_running_loop()
    with ThreadPoolExecutor(max_workers=4) as pool:
        result = await loop.run_in_executor(pool, legacy_call, 10)
        print(result)

asyncio.run(main())
```

이 패턴은 마이그레이션 중에 유용하지만, thread pool 내부에서 다시 event loop를 직접 만지거나 공유 상태를 복잡하게 만들면 디버깅이 어려워집니다.

### Takeaway

혼합 모델은 전환을 부드럽게 해줍니다. 단, event loop와 thread pool의 책임 경계를 명확히 유지해야 합니다.

## Item 78. Maximize Responsiveness of asyncio Event Loops with async-friendly Worker Threads

### Central Question

이벤트 루프가 “멈춘 것처럼” 보이는 원인은 무엇일까?

### My Explanation

`asyncio`에서 가장 흔한 실수는 coroutine 안에서 오래 걸리는 동기 함수를 직접 호출하는 것입니다.

```python
import asyncio
import time

async def bad() -> None:
    time.sleep(1)  # 이벤트 루프 전체를 막음

async def good() -> None:
    await asyncio.sleep(1)  # 다른 task가 실행될 수 있음
```

CPU 작업이나 blocking 라이브러리를 피할 수 없다면 worker thread/process로 보내야 합니다.

```python
async def compress_async(data: bytes) -> bytes:
    import zlib
    return await asyncio.to_thread(zlib.compress, data)
```

또한 긴 task에는 timeout과 cancellation 처리가 필요합니다.

```python
async def fetch_with_timeout() -> None:
    try:
        await asyncio.wait_for(fetch_metadata("x"), timeout=1.0)
    except TimeoutError:
        print("too slow")
```

### Takeaway

async 시스템의 품질은 평균 처리량뿐 아니라 반응성으로 결정됩니다. 이벤트 루프를 막는 코드를 찾아 worker로 분리해야 합니다.

## Item 79. Consider concurrent.futures for True Parallelism

### Central Question

Python에서 CPU-bound 작업을 진짜 병렬로 실행하려면 무엇을 고려해야 할까?

### My Explanation

`concurrent.futures`는 thread, process, interpreter 기반 실행을 비슷한 인터페이스로 다룰 수 있게 해줍니다. CPU-bound 작업에는 일반적으로 `ProcessPoolExecutor`를 먼저 떠올릴 수 있습니다.

```python
from concurrent.futures import ProcessPoolExecutor


def count_primes(limit: int) -> int:
    total = 0
    for number in range(2, limit):
        for divisor in range(2, int(number ** 0.5) + 1):
            if number % divisor == 0:
                break
        else:
            total += 1
    return total

with ProcessPoolExecutor() as executor:
    print(list(executor.map(count_primes, [20_000, 21_000, 22_000])))
```

Python 3.14에서는 `InterpreterPoolExecutor`도 표준 문서에 포함됩니다. 각 worker가 자체 interpreter를 사용하므로 격리와 병렬성의 새로운 절충점이 됩니다. 다만 직렬화 비용, extension module 호환성, 공유 객체 제한을 반드시 검토해야 합니다.

```python
# Python 3.14+
# from concurrent.futures import InterpreterPoolExecutor
# with InterpreterPoolExecutor() as executor:
#     results = list(executor.map(count_primes, [20_000, 21_000]))
```

### Takeaway

진짜 병렬성은 비용이 있습니다. 프로세스와 인터프리터는 GIL 한계를 우회할 수 있지만, 데이터 전달과 초기화 비용이 있으므로 측정으로 판단해야 합니다.

## Chapter Summary

| Decision | Prefer | Avoid |
|---|---|---|
| 외부 프로그램 실행 | `subprocess.run(..., check=True, timeout=...)` | shell 문자열 남용 |
| blocking I/O | thread pool | 요청마다 무제한 thread 생성 |
| high concurrency I/O | `asyncio` | coroutine 안의 blocking call |
| shared state | queue/message passing, 작은 lock 범위 | 전역 mutable state 난립 |
| CPU-bound parallelism | process/interpreter/native code | 스레드만 늘리기 |
