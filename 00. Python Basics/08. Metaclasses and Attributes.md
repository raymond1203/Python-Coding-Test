> 기준 버전: **Python 3.14.6**

## Chapter Thesis

Python의 attribute 시스템은 단순한 field access가 아닙니다. `obj.name`은 내부적으로 lookup protocol, descriptor protocol, class body 실행, metaclass 동작과 연결됩니다.

이 장의 목표는 metaclass를 멋있게 쓰는 것이 아니라, **대부분의 문제를 더 단순한 attribute 도구로 해결하는 법**을 이해하는 것입니다.

## Item 58. Use Plain Attributes Instead of Setter and Getter Methods

### Central Question

Python에서 모든 attribute에 getter/setter를 만들어야 할까?

### My Explanation

Python은 일반적으로 public attribute에서 시작합니다.

```python
class User:
    def __init__(self, name: str) -> None:
        self.name = name

user = User("Ada")
assert user.name == "Ada"
```

Java식 getter/setter를 습관적으로 만들면 호출부가 불필요하게 무거워집니다.

```python
# Python에서는 대체로 과하다.
# user.get_name()
# user.set_name("Grace")
```

나중에 검증이나 계산이 필요해지면 `@property`로 API를 유지하며 확장할 수 있습니다.

### Takeaway

Python에서는 plain public attribute가 기본입니다. 불필요한 getter/setter보다 단순하고 직접적인 접근을 선호합니다.

## Item 59. Consider @property Instead of Refactoring Attributes

### Central Question

이미 public attribute로 노출한 값을 나중에 계산/검증해야 한다면?

### My Explanation

`@property`는 attribute처럼 접근되지만 내부적으로 method를 실행합니다.

```python
class Celsius:
    def __init__(self, value: float) -> None:
        self._value = value

    @property
    def value(self) -> float:
        return self._value

    @value.setter
    def value(self, new_value: float) -> None:
        if new_value < -273.15:
            raise ValueError("below absolute zero")
        self._value = new_value

temp = Celsius(20)
temp.value = 25
assert temp.value == 25
```

호출자는 `temp.value`라는 attribute API를 계속 사용합니다. 내부 구현만 validation을 추가했습니다.

### Takeaway

처음부터 getter/setter를 만들지 않아도 됩니다. 필요해지는 순간 `@property`로 attribute API를 보존하며 확장할 수 있습니다.

## Item 60. Use Descriptors for Reusable @property Methods

### Central Question

여러 attribute에 같은 검증/계산 규칙이 필요하면 어떻게 재사용할까?

### My Explanation

descriptor는 `__get__`, `__set__`, `__delete__` 같은 method로 attribute access를 제어하는 객체입니다.

```python
class PositiveNumber:
    def __set_name__(self, owner, name):
        self.storage_name = f"_{name}"

    def __get__(self, instance, owner):
        if instance is None:
            return self
        return getattr(instance, self.storage_name)

    def __set__(self, instance, value):
        if value <= 0:
            raise ValueError("value must be positive")
        setattr(instance, self.storage_name, value)

class Product:
    price = PositiveNumber()

    def __init__(self, price: int) -> None:
        self.price = price

product = Product(100)
assert product.price == 100
```

`@property`가 한 class 안의 attribute에 좋다면, descriptor는 여러 class나 여러 field에 같은 attribute 규칙을 재사용할 때 좋습니다.

### Takeaway

descriptor는 attribute access를 객체화한 것입니다. 강력하지만 낯설기 때문에, 반복되는 property 규칙이 보일 때만 사용합니다.

## Item 61. Use __getattr__, __getattribute__, and __setattr__ for Lazy Attributes

### Central Question

존재하지 않거나 아직 계산하지 않은 attribute를 어떻게 처리할까?

### My Explanation

`__getattr__`은 일반 attribute lookup이 실패했을 때만 호출됩니다.

```python
class LazyConfig:
    def __init__(self) -> None:
        self.loaded = False

    def __getattr__(self, name: str):
        if name == "settings":
            self.loaded = True
            value = {"debug": True}
            self.settings = value
            return value
        raise AttributeError(name)

config = LazyConfig()
assert config.loaded is False
assert config.settings == {"debug": True}
assert config.loaded is True
```

`__getattribute__`는 모든 attribute access에 개입하므로 훨씬 위험합니다. `__setattr__`는 모든 attribute assignment에 개입합니다.

### Takeaway

lazy attribute에는 먼저 `@property`나 `__getattr__`을 고려합니다. `__getattribute__`는 모든 access에 관여하므로 마지막 수단에 가깝습니다.

## Item 62. Validate Subclasses with __init_subclass__

### Central Question

subclass가 만들어지는 순간 계약을 검사할 수 있을까?

### My Explanation

`__init_subclass__`는 subclass가 정의될 때 호출됩니다.

```python
class RequiresName:
    def __init_subclass__(cls) -> None:
        super().__init_subclass__()
        if not hasattr(cls, "name"):
            raise TypeError("subclasses must define name")

class Plugin(RequiresName):
    name = "example"

assert Plugin.name == "example"
```

이 hook은 metaclass보다 단순하게 subclass validation을 구현할 수 있게 합니다.

### Takeaway

subclass가 특정 class attribute나 method를 가져야 한다면 `__init_subclass__`로 생성 시점에 빠르게 실패하게 만들 수 있습니다.

## Item 63. Register Class Existence with __init_subclass__

### Central Question

새 subclass가 생길 때 자동으로 registry에 등록하려면?

### My Explanation

`__init_subclass__`는 plugin registry에도 사용할 수 있습니다.

```python
class Command:
    registry: dict[str, type["Command"]] = {}

    name: str

    def __init_subclass__(cls) -> None:
        super().__init_subclass__()
        if hasattr(cls, "name"):
            Command.registry[cls.name] = cls

class StartCommand(Command):
    name = "start"

assert Command.registry["start"] is StartCommand
```

class가 정의되는 순간 registry가 갱신됩니다.

### Takeaway

명시적인 등록 호출이 반복된다면 `__init_subclass__`로 class 생성 시점 등록을 고려할 수 있습니다.

## Item 64. Annotate Class Attributes with __set_name__

### Central Question

descriptor는 자신이 어떤 attribute 이름에 할당되었는지 알 수 있을까?

### My Explanation

`__set_name__`은 descriptor가 owner class와 attribute name을 알게 해줍니다.

```python
class LoggedField:
    def __set_name__(self, owner, name):
        self.name = name
        self.storage_name = f"_{name}"

    def __get__(self, instance, owner):
        if instance is None:
            return self
        return getattr(instance, self.storage_name, None)

    def __set__(self, instance, value):
        setattr(instance, self.storage_name, value)

class Account:
    email = LoggedField()

account = Account()
account.email = "ada@example.com"
assert account.email == "ada@example.com"
```

이 기능 덕분에 descriptor를 재사용하면서도 field 이름을 하드코딩하지 않을 수 있습니다.

### Takeaway

`__set_name__`은 descriptor가 class body 안에서 자기 위치를 알게 해주는 hook입니다.

## Item 65. Consider Class Body Definition Order to Establish Relationships Between Attributes

### Central Question

class body는 단순 선언인가, 실행되는 코드인가?

### My Explanation

Python의 class body는 실행됩니다. 따라서 class attribute가 정의된 순서가 의미를 가질 수 있습니다.

```python
class Columns:
    id = "INTEGER"
    name = "TEXT"
    email = "TEXT"

assert list(Columns.__dict__).index("id") < list(Columns.__dict__).index("email")
```

현대 Python에서는 dictionary ordering 덕분에 class namespace의 순서를 활용하는 framework가 많습니다. 예를 들어 field 선언 순서를 schema 순서로 사용할 수 있습니다.

하지만 순서에 의미가 있다면 문서화하고 테스트해야 합니다. 암묵적 순서 의존은 유지보수자를 놀라게 할 수 있습니다.

### Takeaway

class body는 실행되는 코드이며 정의 순서도 관찰될 수 있습니다. 다만 순서가 의미라면 명시적으로 다루어야 합니다.

## Item 66. Prefer Class Decorators over Metaclasses for Composable Class Extensions

### Central Question

class를 생성 후 수정해야 한다면 꼭 metaclass가 필요할까?

### My Explanation

class decorator는 class 객체를 받아 수정하거나 대체한 뒤 반환합니다.

```python
def add_label(cls):
    cls.label = cls.__name__.lower()
    return cls

@add_label
class Report:
    pass

assert Report.label == "report"
```

metaclass는 class 생성 과정을 근본적으로 제어합니다. 강력하지만 여러 metaclass를 조합하기 어렵고 읽는 사람에게 부담을 줍니다.

class decorator는 특정 class 확장에 더 단순하고 조합하기 쉽습니다.

### Takeaway

metaclass는 마지막 수단입니다. class 생성 후 가벼운 확장만 필요하다면 class decorator나 `__init_subclass__`를 먼저 고려합니다.
