## Chapter Thesis

성능 최적화의 첫 번째 원칙은 “빠르게 만들기”가 아니라 **어디가 느린지 틀리지 않기**입니다. Python은 표현력이 높은 언어이기 때문에 알고리즘, 자료구조, I/O 구조만 바꿔도 충분한 경우가 많습니다. 그래도 부족할 때는 네이티브 라이브러리, extension module, zero-copy buffer, lazy import 같은 경계 최적화를 검토합니다.

이 챕터는 Python을 “느리다/빠르다”로 평가하지 않고, **측정 가능한 병목을 발견하고 비용 대비 효과가 큰 개선을 선택하는 능력**을 목표로 합니다.

## Item 92. Profile Before Optimizing

### Central Question

어떤 코드를 먼저 최적화해야 할까?

### My Explanation

직감으로 고른 “느려 보이는 코드”는 실제 병목이 아닐 수 있습니다. `cProfile`은 함수 단위 호출 횟수와 누적 시간을 보여주므로, 먼저 전체 흐름에서 비용이 큰 지점을 찾는 데 좋습니다.

```python
import cProfile
import pstats


def main() -> None:
    data = [str(number) for number in range(100_000)]
    total = sum(int(value) for value in data)
    print(total)

with cProfile.Profile() as profiler:
    main()

stats = pstats.Stats(profiler).sort_stats("cumulative")
stats.print_stats(10)
```

profiling 결과를 볼 때는 다음을 구분합니다.

- 호출 횟수가 많아서 느린가?
- 한 번 호출이 비싸서 느린가?
- I/O 대기인가, CPU 계산인가?
- startup이 문제인가, long-running 처리량이 문제인가?

### Takeaway

최적화는 측정에서 시작합니다. profiling 없이 최적화하면 성능이 아니라 코드 복잡도만 증가할 수 있습니다.

## Item 93. Optimize Performance-Critical Code Using `timeit` Microbenchmarks

### Central Question

작은 코드 조각의 성능을 공정하게 비교하려면?

### My Explanation

`timeit`은 반복 실행과 환경 설정을 통해 작은 표현식의 실행 시간을 비교하는 데 적합합니다.

```python
from timeit import timeit

setup = "numbers = list(range(1000))"

list_comp = timeit("[x * x for x in numbers]", setup=setup, number=10_000)
map_call = timeit("list(map(lambda x: x * x, numbers))", setup=setup, number=10_000)

print(list_comp, map_call)
```

microbenchmark를 해석할 때는 주의해야 합니다.

- 입력 크기가 현실적인가?
- cache나 warm-up 효과가 있는가?
- 전체 프로그램에서 의미 있는 비중인가?
- 가독성 손실을 감수할 만큼 큰 차이인가?

### Takeaway

`timeit`은 작은 선택지를 비교하는 도구입니다. 전체 병목을 찾는 profiling과 함께 사용해야 의미가 있습니다.

## Item 94. Know When and How to Replace Python with Another Programming Language

### Central Question

언제 Python 밖으로 나가야 할까?

### My Explanation

Python을 대체하기 전에 먼저 알고리즘, 자료구조, I/O 구조, 캐싱을 점검해야 합니다. 그래도 CPU-bound hot path가 명확하고, Python 레벨 최적화로 부족하다면 다른 언어 또는 네이티브 라이브러리 연동을 고려합니다.

좋은 분리 기준은 다음과 같습니다.

| Python에 남길 것 | 다른 언어/네이티브로 보낼 후보 |
|---|---|
| orchestration, glue code | tight numeric loop |
| 비즈니스 규칙 | 이미지/신호 처리 kernel |
| 파일/네트워크 흐름 제어 | 압축, 암호화, 벡터화 계산 |
| 테스트와 CLI | 매우 큰 데이터 변환 hot path |

이때 Python API는 유지하고, 내부 구현만 교체하면 사용자 입장에서는 복잡도가 줄어듭니다.

### Takeaway

Python을 버리는 것이 아니라 Python의 강점과 다른 언어의 강점을 나눕니다. 경계를 작게 유지할수록 테스트와 배포가 쉬워집니다.

## Item 95. Consider `ctypes` to Rapidly Integrate with Native Libraries

### Central Question

C 라이브러리를 빠르게 호출해야 한다면 무엇을 사용할 수 있을까?

### My Explanation

`ctypes`는 C ABI를 통해 동적 라이브러리 함수를 Python에서 호출할 수 있게 합니다. 별도의 컴파일된 Python extension을 만들기 전에 빠르게 실험하기 좋습니다.

```python
import ctypes
import ctypes.util

libc_path = ctypes.util.find_library("c")
if libc_path is not None:
    libc = ctypes.CDLL(libc_path)
    libc.abs.argtypes = [ctypes.c_int]
    libc.abs.restype = ctypes.c_int
    assert libc.abs(-3) == 3
```

다만 타입 선언을 잘못하면 crash나 잘못된 값이 발생할 수 있습니다. `argtypes`, `restype`, 메모리 소유권, 문자열 encoding을 명확히 해야 합니다.

### Takeaway

`ctypes`는 네이티브 연동을 빠르게 시작하게 해주지만, Python의 안전한 예외 세계를 벗어나는 순간이기도 합니다. 타입과 메모리 계약을 문서화해야 합니다.

## Item 96. Consider Extension Modules to Maximize Performance and Ergonomics

### Central Question

성능과 사용성을 모두 얻으려면 어떤 구조가 좋을까?

### My Explanation

extension module은 고성능 구현을 Python 모듈처럼 노출합니다. 사용자는 Pythonic API를 쓰고, 내부 hot path만 C/C++/Rust 등으로 실행할 수 있습니다.

좋은 extension 설계는 다음을 지킵니다.

- Python API는 작고 안정적이다.
- 입력 검증과 예외 메시지는 Python 사용자에게 친숙하다.
- 성능이 필요한 부분만 native로 간다.
- fallback Python 구현이나 테스트가 있다.
- wheel 배포, 플랫폼 호환성, CI를 고려한다.

이 선택지는 강력하지만 빌드와 배포 비용이 있습니다. 단순한 과제 풀이 코드나 작은 스크립트에서는 과한 선택일 수 있습니다.

### Takeaway

extension은 마지막 단계의 성능 도구입니다. 먼저 병목을 측정하고, 경계를 작게 잡은 뒤, Python API 품질을 유지해야 합니다.

## Item 97. Rely on Precompiled Bytecode and File System Caching to Improve Startup Time

### Central Question

Python 프로그램의 시작 시간이 느릴 때 무엇을 봐야 할까?

### My Explanation

Python은 모듈을 import할 때 소스 코드를 bytecode로 컴파일하고, 보통 `__pycache__`에 캐시합니다. 같은 환경에서 다시 실행하면 이 캐시 덕분에 startup 비용이 줄 수 있습니다.

startup 최적화에서 확인할 것들입니다.

- import되는 모듈 수가 과도한가?
- CLI 시작 시 필요 없는 무거운 라이브러리를 import하는가?
- 파일 시스템이 느린 환경인가?
- 컨테이너/서버리스처럼 cold start가 중요한가?

`compileall`은 미리 bytecode를 생성하는 데 사용할 수 있습니다.

```bash
python -m compileall my_package
```

### Takeaway

startup 성능은 실행 중 처리량과 다른 문제입니다. import graph와 bytecode cache를 별도로 관찰해야 합니다.

## Item 98. Lazy-Load Modules with Dynamic Imports to Reduce Startup Time

### Central Question

무거운 import를 언제 늦춰야 할까?

### My Explanation

모듈 import는 파일 읽기, 실행, 객체 생성 비용을 가집니다. 특정 기능을 사용할 때만 필요한 무거운 의존성은 함수 내부에서 늦게 import할 수 있습니다.

```python
def render_plot(data: list[float]) -> None:
    import matplotlib.pyplot as plt  # 실제로 plot이 필요할 때만 import

    plt.plot(data)
    plt.show()
```

하지만 lazy import는 의존성을 숨겨서 테스트와 정적 분석을 어렵게 만들 수 있습니다. 기준은 “startup path에 정말 필요한가?”입니다.

더 명시적인 동적 import가 필요하면 `importlib`를 사용할 수 있습니다.

```python
import importlib

json_module = importlib.import_module("json")
print(json_module.dumps({"ok": True}))
```

### Takeaway

lazy import는 startup 최적화 도구입니다. 무분별하게 쓰면 의존성이 흐려지므로, cold start에 영향을 주는 import에만 적용합니다.

## Item 99. Consider `memoryview` and `bytearray` for Zero-Copy Interactions with `bytes`

### Central Question

큰 binary 데이터를 다룰 때 복사를 줄이려면?

### My Explanation

`bytes`는 immutable이므로 slicing하면 새 객체가 만들어질 수 있습니다. 큰 데이터에서 반복 복사가 생기면 메모리와 시간이 낭비됩니다. `memoryview`는 buffer protocol을 통해 같은 메모리를 다른 view로 바라볼 수 있게 합니다.

```python
data = bytearray(b"abcdefghij")
view = memoryview(data)

part = view[2:5]
part[0] = ord("X")

assert data == bytearray(b"abXdefghij")
```

`bytearray`는 mutable binary buffer가 필요할 때 유용하고, `memoryview`는 복사 없이 slice나 protocol 처리를 하고 싶을 때 유용합니다. 단, 원본 buffer의 수명과 mutability를 이해해야 합니다.

### Takeaway

binary hot path에서는 객체 복사도 성능 병목이 됩니다. `memoryview`와 `bytearray`는 Python 레벨에서 zero-copy 사고방식을 익히는 핵심 도구입니다.
