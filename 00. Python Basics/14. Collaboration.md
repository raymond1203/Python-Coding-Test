## Chapter Thesis

좋은 Python 프로젝트는 혼자 실행했을 때만 좋은 코드가 아닙니다. 다른 사람이 설치하고, 읽고, 테스트하고, 확장하고, 안전하게 업그레이드할 수 있어야 합니다. 협업 품질은 `venv`, package 구조, docstring, typing, warnings, custom exception, packaging 같은 표준 습관에서 나옵니다.

이 챕터의 목표는 “GitHub에 올릴 수 있는 코드”가 아니라 **처음 보는 사람이 신뢰하고 따라올 수 있는 Python 프로젝트**를 만드는 기준을 세우는 것입니다.

## Item 116. Know Where to Find Community-Built Modules

### Central Question

외부 라이브러리는 어떻게 고르고 신뢰해야 할까?

### My Explanation

Python 생태계의 중심은 PyPI와 Packaging User Guide입니다. 하지만 “설치 가능하다”와 “프로젝트에 넣어도 된다”는 다릅니다.

외부 모듈을 고를 때는 다음을 확인합니다.

| Check | Why it matters |
|---|---|
| 최근 릴리스/유지보수 | 보안과 호환성 |
| 문서 품질 | 학습/운영 비용 |
| 라이선스 | 공개 repo와 배포 가능성 |
| 의존성 수 | 공급망 위험과 설치 비용 |
| Python 3.14 호환성 | 현재 기준 버전과 맞는지 |
| 테스트/CI 상태 | 안정성 신호 |

작은 기능 하나를 위해 무거운 의존성을 추가하는 것은 장기 유지보수 비용을 늘릴 수 있습니다.

### Takeaway

라이브러리 선택은 기술 선택이자 신뢰 선택입니다. 기능뿐 아니라 유지보수, 라이선스, 호환성, 의존성 비용을 함께 봅니다.

## Item 117. Use Virtual Environments for Isolated and Reproducible Dependencies

### Central Question

왜 프로젝트마다 가상환경을 분리해야 할까?

### My Explanation

`venv`는 프로젝트별 Python 패키지 설치 공간을 분리합니다. 전역 환경에 패키지를 섞으면 프로젝트 간 버전 충돌이 생기고, 다른 사람이 재현하기 어렵습니다.

```bash
python -m venv .venv
.venv\Scripts\activate  # Windows PowerShell에서는 상황에 따라 Activate.ps1 사용
python -m pip install -U pip
```

repo에는 보통 `.venv` 자체를 올리지 않고, 의존성 선언 파일을 올립니다. 프로젝트 성격에 따라 `requirements.txt`, `pyproject.toml`, lock file을 사용합니다.

```text
.venv/        # ignore
pyproject.toml
README.md
src/
tests/
```

### Takeaway

가상환경은 “내 컴퓨터에서는 되는데요”를 줄이는 첫 장치입니다. 실행 환경도 코드의 일부처럼 관리해야 합니다.

## Item 118. Write Docstrings for Every Function, Class, and Module

### Central Question

docstring은 주석과 무엇이 다를까?

### My Explanation

docstring은 객체의 `__doc__`로 노출되는 공식 문서입니다. 모듈, 클래스, 함수의 목적과 사용 계약을 설명합니다.

```python
def parse_score(text: str) -> int:
    """Parse a score string into an integer between 0 and 100.

    Raises:
        ValueError: If the input is not an integer or is out of range.
    """
    value = int(text)
    if not 0 <= value <= 100:
        raise ValueError("score must be between 0 and 100")
    return value
```

좋은 docstring은 구현을 반복하지 않습니다. 대신 호출자가 알아야 할 목적, 인자 의미, 반환값, 예외, 부작용을 설명합니다.

### Takeaway

주석은 구현을 보조하고, docstring은 API 사용자를 돕습니다. 공개 함수와 클래스에는 사용 계약을 남겨야 합니다.

## Item 119. Use Packages to Organize Modules and Provide Stable APIs

### Central Question

파일이 많아질 때 package 구조는 어떻게 잡아야 할까?

### My Explanation

package는 관련 모듈을 묶고, `__init__.py`를 통해 공개 API를 조정할 수 있게 합니다.

```text
src/
  pycoding/
    __init__.py
    parsing.py
    ranking.py
    _internal.py
```

`_internal.py`처럼 underscore가 붙은 모듈은 내부 구현임을 나타내고, `__init__.py`에서는 외부 사용자가 의존해도 되는 이름만 노출할 수 있습니다.

```python
# pycoding/__init__.py
from .parsing import parse_score
from .ranking import rank_scores

__all__ = ["parse_score", "rank_scores"]
```

내부 구조는 바뀔 수 있지만 공개 API는 안정적으로 유지해야 협업자가 안심하고 사용할 수 있습니다.

### Takeaway

package 구조는 폴더 정리가 아니라 API 설계입니다. 내부 구현과 공개 표면을 분리해야 합니다.

## Item 120. Consider Module-Scoped Code to Configure Deployment Environments

### Central Question

모듈 import 시점에 실행되는 코드는 어디까지 허용해야 할까?

### My Explanation

Python 모듈의 top-level 코드는 import될 때 실행됩니다. 상수 정의, logger 설정, 환경 변수 읽기처럼 가벼운 설정은 가능하지만, 네트워크 호출이나 큰 파일 로딩 같은 무거운 작업은 import를 느리고 예측하기 어렵게 만듭니다.

```python
import os

ENVIRONMENT = os.getenv("APP_ENV", "development")
DEBUG = ENVIRONMENT != "production"
```

실행 진입점은 `if __name__ == "__main__":` 아래에 두는 것이 좋습니다.

```python
def main() -> None:
    print("run application")

if __name__ == "__main__":
    main()
```

### Takeaway

import는 가능한 한 가볍고 안전해야 합니다. 배포 환경 설정은 명시적으로 읽되, 실행 부작용은 진입점으로 분리합니다.

## Item 121. Define a Root `Exception` to Insulate Callers from APIs

### Central Question

라이브러리 사용자가 어떤 예외를 잡아야 할지 어떻게 알 수 있을까?

### My Explanation

프로젝트별 root exception을 정의하면 호출자가 라이브러리 내부 예외를 안정적으로 처리할 수 있습니다.

```python
class CodingRepoError(Exception):
    """Base exception for this package."""

class ParseError(CodingRepoError):
    pass

class RankingError(CodingRepoError):
    pass
```

내부에서 `ValueError`, `OSError` 같은 낮은 수준 예외가 발생하더라도 API 경계에서는 프로젝트 예외로 변환할 수 있습니다.

```python
def parse_score(text: str) -> int:
    try:
        return int(text)
    except ValueError as error:
        raise ParseError(f"invalid score: {text!r}") from error
```

### Takeaway

root exception은 사용자에게 “이 라이브러리에서 온 실패”를 잡을 수 있는 안정적 경계를 제공합니다.

## Item 122. Know How to Break Circular Dependencies

### Central Question

두 모듈이 서로 import해서 오류가 날 때 어떻게 해결할까?

### My Explanation

circular dependency는 보통 책임이 섞였다는 신호입니다.

```text
parser.py  -> models.py
models.py  -> parser.py
```

해결 방법은 여러 가지입니다.

1. 공통 타입/상수를 별도 모듈로 분리한다.
2. 높은 수준 모듈이 낮은 수준 모듈을 의존하게 단방향 그래프를 만든다.
3. import를 함수 내부로 늦추되, 구조적 해결이 가능한지 먼저 본다.
4. typing 전용 import는 `TYPE_CHECKING`으로 분리한다.

```python
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from .models import User
```

### Takeaway

순환 import는 “import 위치” 문제가 아니라 “모듈 책임” 문제인 경우가 많습니다. 의존성 방향을 단순하게 만들어야 합니다.

## Item 123. Consider `warnings` to Refactor and Migrate Usage

### Central Question

기존 API를 바꿔야 할 때 사용자를 어떻게 보호할까?

### My Explanation

공개 API를 바로 제거하면 사용자 코드가 갑자기 깨집니다. `warnings`는 변경 예정 사실을 런타임에 알려주고, migration window를 제공합니다.

```python
import warnings


def old_parse(text: str) -> int:
    warnings.warn(
        "old_parse() is deprecated; use parse_score() instead",
        DeprecationWarning,
        stacklevel=2,
    )
    return parse_score(text)
```

`stacklevel=2`는 경고 위치가 라이브러리 내부가 아니라 호출자 코드에 가깝게 보이도록 도와줍니다. 테스트에서는 deprecation warning이 의도대로 발생하는지도 확인할 수 있습니다.

### Takeaway

경고는 협업자를 배려하는 변경 관리 도구입니다. 공개 API를 바꿀 때는 제거보다 먼저 경고와 대안을 제공합니다.

## Item 124. Consider Static Analysis via `typing` to Obviate Bugs

### Central Question

타입 힌트는 실행 속도를 높이기 위한 것일까?

### My Explanation

Python의 타입 힌트는 주로 정적 분석, IDE 지원, 문서화를 위한 계약입니다. 런타임에서 자동으로 강제되지 않지만, 함수의 의도를 명확히 하고 많은 실수를 사전에 잡게 해줍니다.

```python
from collections.abc import Iterable


def average(numbers: Iterable[float]) -> float:
    values = list(numbers)
    if not values:
        raise ValueError("empty input")
    return sum(values) / len(values)
```

Python 3.14에서는 annotation 처리 방식이 발전했으므로, 타입 힌트를 런타임 로직에 과도하게 의존하기보다 정적 분석과 문서화 관점에서 안정적으로 사용하는 태도가 중요합니다.

타입 설계의 좋은 기준은 다음과 같습니다.

- concrete class보다 protocol/abstract collection을 받을 수 있는가?
- `None` 가능성을 명시했는가?
- 반환 타입이 예외 가능성과 함께 이해되는가?
- public API의 타입이 내부 구현을 노출하지 않는가?

### Takeaway

타입 힌트는 Python을 정적 언어로 바꾸는 장치가 아니라, 협업자가 코드를 안전하게 이해하도록 돕는 계약입니다.

## Item 125. Prefer Open Source Projects for Bundling Python Programs over `zipimport` and `zipapp`

### Central Question

Python 프로그램을 하나로 묶어 배포할 때 무엇을 고려해야 할까?

### My Explanation

`zipimport`와 `zipapp`은 Python 코드를 zip archive에서 실행하게 해주는 표준 기능입니다. 간단한 내부 도구에는 유용할 수 있지만, 복잡한 의존성, native extension, platform-specific 파일, 리소스 패키징이 들어가면 한계가 빨리 드러납니다.

```bash
python -m zipapp my_app -m "my_app:main"
```

공개 배포나 포트폴리오 프로젝트에서는 표준 packaging 흐름과 널리 쓰이는 오픈소스 도구를 검토하는 편이 유지보수에 좋습니다.

| Goal | Candidate |
|---|---|
| library 배포 | `pyproject.toml`, wheel |
| CLI 배포 | console scripts entry point |
| 독립 실행 파일 | PyInstaller 등 별도 bundler 검토 |
| 의존성 재현 | lock file, virtual environment |

### Takeaway

표준 라이브러리의 packaging 기능은 기본기를 이해하는 데 좋지만, 실제 배포 요구사항은 더 넓습니다. 공개 프로젝트라면 생태계 표준 도구와 문서를 우선 검토합니다.
