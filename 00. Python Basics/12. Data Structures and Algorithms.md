## Item 100. Sort by Complex Criteria Using the `key` Parameter

### Central Question

복잡한 기준으로 정렬할 때 Pythonic한 방식은 무엇일까?

### My Explanation

Python 정렬은 `key` 함수를 통해 각 원소에서 비교에 사용할 값을 추출합니다. 비교 함수를 매번 호출하는 방식보다 key를 한 번 계산하는 방식이 명확하고 효율적입니다.

```python
students = [
    {"name": "Ada", "score": 95, "age": 17},
    {"name": "Grace", "score": 95, "age": 16},
    {"name": "Linus", "score": 88, "age": 18},
]

ranked = sorted(students, key=lambda s: (-s["score"], s["age"], s["name"]))
```

tuple key는 왼쪽부터 순서대로 비교됩니다. 위 예시는 점수 내림차순, 나이 오름차순, 이름 오름차순의 기준을 한 줄에 표현합니다.

`operator.itemgetter`나 `attrgetter`를 쓰면 반복되는 lambda를 더 명확하게 만들 수 있습니다.

```python
from operator import itemgetter

by_name = sorted(students, key=itemgetter("name"))
```

### Takeaway

정렬 기준은 `key`로 명명합니다. 복잡한 기준일수록 tuple key를 사용해 우선순위를 데이터처럼 드러냅니다.

## Item 101. Know the Difference Between `sort` and `sorted`

### Central Question

`list.sort()`와 `sorted()`는 언제 다르게 써야 할까?

### My Explanation

`list.sort()`는 리스트를 제자리에서 변경하고 `None`을 반환합니다. `sorted()`는 iterable을 받아 새 list를 반환합니다.

```python
numbers = [3, 1, 2]
returned = numbers.sort()
assert returned is None
assert numbers == [1, 2, 3]

original = [3, 1, 2]
ordered = sorted(original)
assert original == [3, 1, 2]
assert ordered == [1, 2, 3]
```

선택 기준은 소유권입니다.

| 상황 | 선택 |
|---|---|
| 내가 가진 리스트를 바꿔도 된다 | `list.sort()` |
| 원본 순서를 보존해야 한다 | `sorted()` |
| iterable이 list가 아니다 | `sorted()` |
| 메모리 추가 할당을 줄이고 싶다 | `list.sort()` |

Python 정렬은 stable sort입니다. 같은 key를 가진 원소의 기존 순서가 보존되므로, 다단계 정렬을 안전하게 구성할 수 있습니다.

### Takeaway

`sort`는 mutation이고 `sorted`는 transformation입니다. 원본 데이터의 소유권과 이후 사용 여부를 기준으로 선택합니다.

## Item 102. Consider Searching Sorted Sequences with `bisect`

### Central Question

정렬된 리스트에서 삽입 위치나 구간 경계를 빠르게 찾으려면?

### My Explanation

`bisect`는 정렬된 sequence에서 이진 탐색으로 삽입 위치를 찾습니다. 리스트 자체의 중간 삽입은 O(n)이지만, 위치 탐색은 O(log n)입니다.

```python
from bisect import bisect_left, bisect_right

scores = [60, 70, 70, 80, 90]

left = bisect_left(scores, 70)
right = bisect_right(scores, 70)
assert (left, right) == (1, 3)
assert scores[left:right] == [70, 70]
```

`bisect_left`는 같은 값의 왼쪽 경계, `bisect_right`는 오른쪽 경계를 찾습니다. 이 차이를 알면 lower bound / upper bound 사고를 Python에서 자연스럽게 표현할 수 있습니다.

```python
from bisect import insort

numbers = [1, 3, 5]
insort(numbers, 4)
assert numbers == [1, 3, 4, 5]
```

### Takeaway

`bisect`의 전제는 “정렬된 상태가 계속 유지된다”는 불변식입니다. 이 불변식이 깨지면 결과도 의미가 없어집니다.

## Item 103. Prefer `deque` for Producer-Consumer Queues

### Central Question

왜 리스트보다 `deque`가 큐에 적합할까?

### My Explanation

`list.append`와 `list.pop()`은 오른쪽 끝에서는 빠르지만, 왼쪽 끝에서 `pop(0)`을 하면 모든 원소를 이동해야 하므로 O(n)입니다. `collections.deque`는 양쪽 끝 삽입/삭제에 적합합니다.

```python
from collections import deque

queue: deque[str] = deque()
queue.append("job-1")
queue.append("job-2")

assert queue.popleft() == "job-1"
assert queue.popleft() == "job-2"
```

`maxlen`을 지정하면 고정 길이 ring buffer처럼 사용할 수 있습니다.

```python
recent = deque(maxlen=3)
for value in [1, 2, 3, 4]:
    recent.append(value)
assert list(recent) == [2, 3, 4]
```

thread 간 안전한 작업 큐가 필요하면 `queue.Queue`가 더 적합하지만, 단일 프로세스 내부 자료구조로는 `deque`가 간단하고 빠릅니다.

### Takeaway

FIFO/LIFO 양끝 조작이 핵심이면 `deque`를 먼저 고려합니다. 리스트는 random access와 오른쪽 append에 강합니다.

## Item 104. Know How to Use `heapq` for Priority Queues

### Central Question

우선순위가 가장 높은 작업을 반복해서 꺼내려면?

### My Explanation

`heapq`는 list 위에 heap invariant를 유지합니다. 기본은 min-heap입니다.

```python
from heapq import heappop, heappush

heap: list[tuple[int, str]] = []
heappush(heap, (2, "write docs"))
heappush(heap, (1, "fix bug"))

assert heappop(heap) == (1, "fix bug")
```

동일 priority에서 비교 불가능한 객체가 들어가면 문제가 생길 수 있으므로 tie-breaker를 추가합니다.

```python
from itertools import count

counter = count()
heap: list[tuple[int, int, str]] = []
heappush(heap, (1, next(counter), "task-a"))
heappush(heap, (1, next(counter), "task-b"))
```

Python 3.14에서는 max-heap 전용 API도 추가되었습니다. 이전에는 priority에 음수를 붙이는 방식이 흔했지만, 이제 `heapify_max`, `heappush_max`, `heappop_max` 같은 이름으로 의도를 더 직접 표현할 수 있습니다.

```python
# Python 3.14+
from heapq import heapify_max, heappop_max

numbers = [3, 1, 5, 2]
heapify_max(numbers)
assert heappop_max(numbers) == 5
```

### Takeaway

heap은 “항상 전체가 정렬되어 있다”가 아니라 “루트가 최솟값/최댓값이라는 불변식”을 유지합니다. priority와 tie-breaker를 함께 설계해야 합니다.

## Item 105. Use `datetime` Instead of `time` for Local Clocks

### Central Question

사람이 이해하는 날짜와 시간을 다룰 때 무엇을 써야 할까?

### My Explanation

`time` 모듈은 timestamp, sleep, monotonic clock처럼 시스템 시간과 측정에 가깝습니다. 반면 `datetime`은 달력 날짜, 시간대, 기간 계산을 표현하는 데 적합합니다.

```python
from datetime import datetime, timedelta, timezone

now = datetime.now(timezone.utc)
tomorrow = now + timedelta(days=1)
print(tomorrow.isoformat())
```

local time을 다룰 때는 naive datetime과 aware datetime을 구분해야 합니다.

- naive: timezone 정보가 없음
- aware: timezone 정보가 있음

로그, API, 저장소에는 가능하면 UTC aware datetime을 사용하고, 사용자 표시 시점에 local timezone으로 변환하는 구조가 안전합니다.

### Takeaway

시간은 숫자가 아니라 맥락입니다. 달력 의미와 timezone이 필요하면 `datetime`을 사용하고, elapsed time 측정에는 `time.monotonic()` 계열을 고려합니다.

## Item 106. Use `decimal` When Precision Is Paramount

### Central Question

왜 돈 계산에 float를 쓰면 위험할까?

### My Explanation

`float`는 이진 부동소수점이므로 0.1 같은 십진수를 정확히 표현하지 못합니다.

```python
assert 0.1 + 0.2 != 0.3
```

금액, 회계, 정밀한 십진 반올림 규칙이 중요한 경우 `decimal.Decimal`을 사용합니다.

```python
from decimal import Decimal, ROUND_HALF_UP

price = Decimal("19.99")
tax = Decimal("0.0825")
total = price * (Decimal("1") + tax)
rounded = total.quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)
print(rounded)
```

중요한 습관은 `Decimal(0.1)`처럼 float에서 만들지 않고, 문자열에서 만드는 것입니다.

### Takeaway

정밀도가 요구사항이면 숫자 타입도 요구사항입니다. 금액과 명시적 반올림에는 `Decimal`을 사용합니다.

## Item 107. Make `pickle` Serialization Maintainable with `copyreg`

### Central Question

객체를 저장한 뒤 클래스 구조가 바뀌면 어떻게 유지보수할까?

### My Explanation

`pickle`은 Python 객체를 직렬화할 수 있지만, 장기 저장 포맷으로 사용할 때는 클래스 경로, 필드 변경, 버전 호환성 문제가 생깁니다. `copyreg`는 특정 타입의 직렬화 방식을 등록해 객체 복원 로직을 분리할 수 있게 합니다.

```python
import copyreg
import pickle
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int


def pickle_point(point: Point):
    return unpickle_point, (point.x, point.y, 1)


def unpickle_point(x: int, y: int, version: int) -> Point:
    return Point(x=x, y=y)

copyreg.pickle(Point, pickle_point)

payload = pickle.dumps(Point(1, 2))
assert pickle.loads(payload) == Point(1, 2)
```

보안 관점도 중요합니다. 신뢰할 수 없는 pickle 데이터를 역직렬화하면 임의 코드 실행 위험이 있습니다. 외부 입력에는 JSON 같은 안전한 포맷을 우선 고려합니다.

### Takeaway

`pickle`은 Python 내부 객체 그래프 저장에는 강력하지만, 공개 데이터 교환 포맷은 아닙니다. 장기 저장에는 버전과 복원 함수를 설계해야 합니다.
