## Chapter Thesis

**Pythonic Thinking**은 “짧게 쓰는 기술”이 아니라, Python이 코드를 읽고 실행하는 방식에 맞춰 생각하는 태도입니다.

Python다운 코드는 보통 다음 특징을 가집니다.

1. **읽는 사람이 먼저 이해한다.**  
   기계가 실행할 수 있는 코드는 누구나 쓸 수 있습니다. 좋은 Python 코드는 사람이 읽을 때 의도가 먼저 보입니다.

2. **값의 흐름이 드러난다.**  
   Python은 표현식이 강력하지만, 표현식이 길어질수록 이름 붙일 기회를 잃습니다. 이름은 프로그램 안에서 가장 작은 문서입니다.

3. **런타임 언어라는 사실을 인정한다.**  
   Python은 많은 오류를 실행 시점에 발견합니다. 따라서 “작성했으니 맞다”가 아니라 “실행 경로를 검증했는가?”가 중요합니다.

4. **문법보다 의도를 우선한다.**  
   최신 문법을 사용할 수 있다고 항상 좋은 코드는 아닙니다. `match`, `:=`, conditional expression 모두 의도를 더 선명하게 할 때만 사용합니다.

## Item 1. Know Which Version of Python You're Using

### Central Question

내가 작성한 코드는 **어떤 Python 언어 버전과 어떤 구현체** 위에서 실행되는가?

### My Explanation

Python 코드는 “Python”이라는 추상적인 언어에서 실행되는 것처럼 보이지만, 실제로는 특정 버전과 특정 구현체 위에서 실행됩니다.

- 언어 버전: Python 3.14.6, Python 3.13.x 등
- 구현체: CPython, PyPy 등
- 실행 환경: 로컬, 가상환경, 서버, CI, 노트북, 온라인 인터프리터 등

이 셋은 같지 않습니다.

예를 들어 Python 3.14.6 문서를 기준으로 작성한 코드는 Python 3.10 환경에서 실행되지 않을 수 있습니다. 반대로 같은 Python 3.14라도 CPython 구현 세부사항에 의존한 코드는 다른 구현체에서 다르게 동작할 수 있습니다.

Python을 깊게 이해한다는 것은 “이 문법이 된다”에서 끝나지 않고, **이 코드가 어떤 계약 위에서 성립하는지**를 아는 것입니다.

### Python 3.14.6 Verification

Python 공식 문서는 `sys.version_info`를 버전 확인에 적합한 구조로 설명합니다. 문자열인 `sys.version`을 파싱하기보다 구조화된 값을 사용하는 것이 안전합니다.

```python
import sys

assert sys.version_info.major == 3
assert sys.version_info.minor == 14
assert sys.version_info.micro >= 0

print(sys.version_info)
```

버전을 확인할 때는 보통 문자열 비교를 피합니다.

```python
# Bad: 문자열 비교는 의도가 불분명하고 실수하기 쉽다.
version = "3.14.6"
assert version > "3.9.0"  # 우연히 맞아 보일 수 있지만 일반화하기 어렵다.

# Better: 구조화된 tuple 비교를 사용한다.
import sys
assert sys.version_info >= (3, 14)
```

### Language Version vs Implementation

`sys.version_info`는 현재 인터프리터가 따르는 Python 언어 버전을 나타냅니다. `sys.implementation`은 구현체에 대한 정보를 나타냅니다.

```python
import sys

print(sys.version_info)          # Python 언어 버전
print(sys.implementation.name)   # 일반적으로 'cpython'
print(sys.implementation.version)
```

대부분의 일상적인 Python 코드는 CPython에서 실행되지만, 좋은 문서는 “이 내용이 Python 언어의 일반 규칙인지, CPython 구현 세부사항인지”를 구분해야 합니다.

### Common Misunderstandings

- “내 컴퓨터에서 돌아가면 Python에서 돌아간다.”  
  → 실제로는 특정 버전, 특정 구현체, 특정 dependency 조합에서 돌아간 것입니다.

- “최신 문법을 쓰면 더 좋은 코드다.”  
  → 최신 문법은 의도를 선명하게 할 때만 좋은 선택입니다.

- “Python 3이면 다 비슷하다.”  
  → 3.8의 `:=`, 3.10의 `match`, 3.14의 annotation 변화처럼 버전별 차이는 실제 코드 작성에 영향을 줍니다.

### Takeaway

Python 코드의 첫 번째 문맥은 버전입니다. 이 레포는 **Python 3.14.6**을 기준으로 하되, 버전 의존적인 내용은 명시합니다.

## Item 2. Follow the PEP 8 Style Guide

### Central Question

스타일은 취향인가, 아니면 협업과 이해를 위한 약속인가?

### My Explanation

PEP 8은 단순한 미적 규칙이 아닙니다. Python 코드가 여러 사람에게 비슷한 방식으로 읽히도록 만드는 **공통 문법 밖의 문법**입니다.

문법적으로 같은 코드라도 스타일이 다르면 읽는 비용이 달라집니다.

```python
# Hard to scan
result=[x*x for x in range(10)if x%2==0]

# Easier to scan
result = [x * x for x in range(10) if x % 2 == 0]
```

두 코드는 같은 결과를 만들지만, 두 번째 코드는 눈이 구조를 빠르게 찾을 수 있습니다.

Python다운 스타일은 “예쁘게 꾸미기”가 아니라 **읽는 사람의 인지 부하를 낮추는 것**입니다.

### PEP 8의 핵심 태도

PEP 8에서 가장 중요한 부분은 특정 줄 길이보다도 “일관성의 우선순위”입니다.

1. 프로젝트 안에서 일관적이어야 한다.
2. 모듈 안에서 일관적이어야 한다.
3. 함수 안에서 일관적이어야 한다.
4. 하지만 무조건적인 일관성보다 실용적 판단이 우선할 때도 있다.

즉, PEP 8은 기계적인 규칙표가 아니라 판단 기준입니다.

### Naming as Documentation

Python에서는 이름이 매우 중요합니다. 타입 선언이 상대적으로 가볍기 때문에, 이름이 코드의 의도를 많이 설명합니다.

```python
# Poor
x = [p for p in people if p.age >= 20]

# Better
adults = [person for person in people if person.age >= 20]
```

`x`도 실행은 됩니다. 그러나 `adults`는 값을 설명합니다. 좋은 이름은 주석을 줄이고, 주석보다 오래 살아남습니다.

### Import Style

Python 파일의 상단 import는 문서의 목차와 비슷합니다. 이 파일이 어떤 세계와 연결되는지 보여줍니다.

```python
# Prefer standard library, third-party, local imports in separated groups.
import json
from pathlib import Path

import requests

from project.models import User
```

### Common Misunderstandings

- “PEP 8을 따르면 자동으로 좋은 코드다.”  
  → PEP 8은 최소한의 읽기 규칙입니다. 구조가 나쁜 코드는 PEP 8을 지켜도 어렵습니다.

- “포매터가 있으니 스타일은 몰라도 된다.”  
  → `black`, `ruff` 같은 도구는 도움을 주지만, 이름 짓기와 구조화는 사람이 해야 합니다.

- “짧은 이름이 Python답다.”  
  → 짧은 범위의 관용적 변수는 괜찮지만, 의미 있는 값에는 의미 있는 이름이 필요합니다.

### Takeaway

PEP 8은 Python 커뮤니티가 합의한 읽기 비용 절감 장치입니다. 스타일은 취향보다 신뢰에 가깝습니다.

## Item 3. Never Expect Python to Detect Errors at Compile Time

### Central Question

Python은 코드를 실행하기 전에 얼마나 많은 오류를 잡아줄까?

### My Explanation

Python은 실행 전에 소스 코드를 파싱하고 바이트코드로 컴파일합니다. 그래서 문법 오류는 비교적 일찍 잡힙니다.

```python
# SyntaxError: 닫는 괄호가 없다.
# print("hello"
```

하지만 Python의 많은 오류는 실제 실행 경로에 도달해야 드러납니다.

```python
def greet(name):
    return f"Hello, {naem}"  # 오타지만 함수 정의 시점에는 실행되지 않는다.

# 여기까지는 조용하다.
# 실제로 호출해야 NameError가 발생한다.
greet("Python")
```

이것은 Python이 “느슨하다”는 뜻이 아닙니다. Python은 동적 언어입니다. 이름, 속성, 타입 관계 중 상당수가 런타임에 결정됩니다.

### Runtime Model

Python에서 다음 오류들은 보통 실행 중에 발견됩니다.

- 존재하지 않는 이름 사용: `NameError`
- 존재하지 않는 속성 접근: `AttributeError`
- 지원하지 않는 연산: `TypeError`
- 값의 형태가 잘못됨: `ValueError`
- 특정 분기에서만 발생하는 오류

```python
def normalize(value):
    if isinstance(value, str):
        return value.strip().lower()
    return value.strip()  # str이 아니면 여기서 AttributeError 가능
```

함수를 정의했다는 사실은 함수가 안전하다는 증거가 아닙니다. 각 실행 경로를 실제로 통과해야 합니다.

### Practical Response

Python 코드를 신뢰할 수 있게 만들려면 다음 습관이 필요합니다.

1. 작은 함수로 나누어 실행 경로를 줄인다.
2. 중요한 분기는 직접 테스트한다.
3. 타입 힌트와 정적 분석 도구를 보조 수단으로 사용한다.
4. 예외 메시지를 읽고, 실패 지점이 아니라 실패 원인을 찾는다.

```python
def divide(total: float, count: int) -> float:
    if count == 0:
        raise ValueError("count must not be zero")
    return total / count

assert divide(10, 2) == 5
```

타입 힌트는 Python 런타임을 정적 언어로 바꾸지 않습니다. 하지만 사람과 도구가 오류 가능성을 더 빨리 발견하도록 돕습니다.

### Common Misunderstandings

- “Python은 컴파일이 없으니 아무 검사를 안 한다.”  
  → Python도 파싱과 바이트코드 컴파일을 합니다. 다만 많은 의미 검사가 런타임에 일어납니다.

- “타입 힌트를 쓰면 런타임 타입 검사가 된다.”  
  → 기본적으로 타입 힌트는 힌트입니다. 별도 도구나 런타임 검사 없이는 강제되지 않습니다.

- “테스트는 큰 프로젝트에서나 필요하다.”  
  → 동적 언어에서는 작은 코드라도 실행 경로 확인이 매우 중요합니다.

### Takeaway

Python 코드는 작성 시점보다 실행 시점에 더 많은 진실을 드러냅니다. 그래서 좋은 Python 개발자는 테스트와 실행 가능한 예제를 문서의 일부로 봅니다.

## Item 4. Write Helper Functions Instead of Complex Expressions

### Central Question

한 줄로 쓸 수 있는 코드는 언제 여러 줄로 나누어야 할까?

### My Explanation

Python은 표현식이 강력합니다. list comprehension, conditional expression, unpacking, assignment expression을 조합하면 많은 일을 한 줄에 할 수 있습니다.

하지만 한 줄에 너무 많은 의미가 들어가면 코드는 “짧지만 어려운 코드”가 됩니다.

```python
# Hard to understand
score = min(100, max(0, int(raw.strip() or 0))) if raw is not None else 0
```

이 코드는 짧지만 여러 개념이 섞여 있습니다.

- `None` 처리
- 공백 제거
- 빈 문자열 기본값
- 정수 변환
- 범위 제한

이럴 때 helper function은 단순한 분리가 아닙니다. **개념에 이름을 붙이는 작업**입니다.

```python
def clamp(value: int, lower: int, upper: int) -> int:
    return min(upper, max(lower, value))


def parse_score(raw: str | None) -> int:
    if raw is None:
        return 0

    text = raw.strip()
    if not text:
        return 0

    return clamp(int(text), 0, 100)

assert parse_score(" 42 ") == 42
assert parse_score("") == 0
assert parse_score(None) == 0
assert parse_score("120") == 100
```

좋은 helper function은 코드 줄 수를 줄이기 위해 존재하지 않습니다. 읽는 사람이 “아, 이 단계는 점수 파싱이구나”라고 이해하게 만듭니다.

### When to Extract

다음 중 하나라도 해당하면 helper function을 고려합니다.

- 표현식 안에 서로 다른 개념이 2개 이상 섞여 있다.
- 같은 계산이 반복된다.
- 조건이 중첩되어 읽는 순서가 어렵다.
- 중간 결과에 이름을 붙이면 의도가 선명해진다.
- 테스트하고 싶은 단위가 보인다.

### Naming the Intermediate Thought

```python
# Before
users = [u for u in users if u.is_active and not u.is_blocked and u.last_login is not None]

# After
def can_receive_notification(user) -> bool:
    return user.is_active and not user.is_blocked and user.last_login is not None

users = [user for user in users if can_receive_notification(user)]
```

두 번째 코드는 조금 길지만, 정책이 이름으로 분리되어 있습니다. 이 이름은 나중에 요구사항이 바뀔 때 수정 지점이 됩니다.

### Common Misunderstandings

- “함수가 많으면 코드가 복잡해진다.”  
  → 의미 없이 쪼개면 복잡해지지만, 개념 단위로 나누면 오히려 단순해집니다.

- “한 줄 코드가 Python답다.”  
  → Python다운 코드는 짧은 코드가 아니라 의도가 드러나는 코드입니다.

- “helper function은 재사용할 때만 만든다.”  
  → 재사용되지 않아도, 이름 붙일 가치가 있으면 만들 수 있습니다.

### Takeaway

복잡한 표현식을 helper function으로 바꾸는 것은 코드를 늘리는 일이 아니라 생각을 정리하는 일입니다.

## Item 5. Prefer Multiple-Assignment Unpacking over Indexing

### Central Question

시퀀스에서 값을 꺼낼 때, 왜 `x[0]`, `x[1]`보다 unpacking이 더 읽기 쉬울까?

### My Explanation

Python의 unpacking은 데이터의 구조를 코드에 직접 드러냅니다.

```python
point = (3, 4)

# Indexing
x = point[0]
y = point[1]

# Unpacking
x, y = point
```

두 방식은 같은 값을 얻습니다. 그러나 unpacking은 “이 값은 두 부분으로 이루어진 좌표다”라는 구조를 보여줍니다.

### Unpacking Is Structural

```python
name, age, city = ("Ada", 36, "London")

assert name == "Ada"
assert age == 36
assert city == "London"
```

unpacking은 길이가 맞지 않으면 즉시 실패합니다.

```python
# ValueError: not enough values to unpack
# name, age, city = ("Ada", 36)
```

이 실패는 오히려 장점입니다. 데이터 구조에 대한 가정이 틀렸음을 빠르게 알려줍니다.

### Swapping Without Temporary Variables

Python에서는 값 교환도 unpacking으로 표현합니다.

```python
a = 10
b = 20

a, b = b, a

assert a == 20
assert b == 10
```

오른쪽이 먼저 평가되고, 그 결과가 왼쪽 이름들에 할당됩니다. 이 동작을 이해하면 Python의 assignment가 단순 대입보다 구조적인 연산임을 알 수 있습니다.

### Starred Unpacking

길이가 유동적인 데이터에서는 starred expression을 사용할 수 있습니다.

```python
first, *middle, last = [1, 2, 3, 4, 5]

assert first == 1
assert middle == [2, 3, 4]
assert last == 5
```

이것은 slicing보다 의도가 분명할 때가 많습니다.

```python
# Indexing/slicing version
items = [1, 2, 3, 4, 5]
first = items[0]
middle = items[1:-1]
last = items[-1]

# Structural version
first, *middle, last = items
```

### Common Misunderstandings

- “unpacking은 tuple에만 쓴다.”  
  → list, tuple, string, iterator 등 iterable이면 사용할 수 있습니다.

- “unpacking은 짧은 코드용 문법이다.”  
  → 핵심은 길이가 아니라 구조를 드러내는 것입니다.

- “starred unpacking은 항상 좋은 선택이다.”  
  → 너무 복잡한 unpacking은 오히려 읽기 어렵습니다. 구조가 직관적일 때만 사용합니다.

### Takeaway

indexing은 위치를 말하고, unpacking은 구조를 말합니다. Python다운 코드는 가능한 한 구조를 드러냅니다.

## Item 6. Always Surround Single-Element Tuples with Parentheses

### Central Question

tuple을 만드는 것은 괄호일까, comma일까?

### My Explanation

Python에서 tuple을 만드는 핵심은 괄호가 아니라 comma입니다.

```python
not_a_tuple = (1)
single_item_tuple = (1,)

assert not_a_tuple == 1
assert single_item_tuple == (1,)
assert isinstance(single_item_tuple, tuple)
```

괄호는 grouping에도 사용됩니다.

```python
value = (1 + 2) * 3
assert value == 9
```

따라서 `(1)`은 “tuple”이 아니라 “괄호로 묶은 정수 1”입니다. 하나짜리 tuple에는 반드시 comma가 필요합니다.

### Why Parentheses Still Matter

문법적으로는 comma가 핵심이지만, 사람이 읽을 때는 괄호가 의도를 분명하게 합니다.

```python
# Works, but easy to miss
tuple_without_parentheses = 1,

# Clearer
single_item_tuple = (1,)
```

특히 함수 반환값이나 인자에서 실수를 줄이는 데 중요합니다.

```python
def get_status():
    return ("ok",)

status = get_status()
assert status == ("ok",)
```

### Common Misunderstandings

- “괄호가 있으면 tuple이다.”  
  → `(1)`은 `int`입니다.

- “comma 하나는 사소한 스타일 문제다.”  
  → single-element tuple에서는 의미를 바꾸는 문법입니다.

- “괄호 없이 쓰는 게 더 Python답다.”  
  → 가능하더라도 의도가 흐려지면 좋은 선택이 아닙니다.

### Takeaway

Python 문법에서 괄호는 종종 grouping이고, comma가 구조를 만듭니다. single-element tuple은 `(value,)`로 명시합니다.

## Item 7. Consider Conditional Expressions for Simple Inline Logic

### Central Question

`if` 문과 conditional expression은 언제 다르게 사용해야 할까?

### My Explanation

conditional expression은 값을 선택하는 표현식입니다.

```python
label = "adult" if age >= 20 else "minor"
```

일반적인 `if` statement는 실행 흐름을 나누는 문장입니다.

```python
if age >= 20:
    label = "adult"
else:
    label = "minor"
```

둘은 비슷해 보이지만 역할이 다릅니다.

- conditional expression: 두 값 중 하나를 선택한다.
- `if` statement: 서로 다른 작업 흐름을 실행한다.

### Good Use

conditional expression은 분기가 짧고 대칭적일 때 좋습니다.

```python
def display_name(name: str | None) -> str:
    return name if name else "Anonymous"

assert display_name("Guido") == "Guido"
assert display_name("") == "Anonymous"
```

### Bad Use

분기가 복잡하거나 부작용이 있으면 일반 `if` 문이 낫습니다.

```python
# Hard to read
result = send_email(user) if user.email_verified else log_unverified_user(user)

# Better
if user.email_verified:
    result = send_email(user)
else:
    result = log_unverified_user(user)
```

conditional expression은 값을 고르는 데 적합하지, 행동을 숨기는 데 적합하지 않습니다.

### Evaluation Order

`x if condition else y`는 condition을 먼저 평가합니다. condition이 참이면 `x`만 평가하고, 거짓이면 `y`만 평가합니다.

```python
def left():
    return "left"


def right():
    raise RuntimeError("right should not be evaluated")

value = left() if True else right()
assert value == "left"
```

### Common Misunderstandings

- “짧으면 conditional expression을 써야 한다.”  
  → 짧음보다 명확성이 우선입니다.

- “중첩 conditional expression도 괜찮다.”  
  → 중첩되면 읽는 순서가 급격히 어려워집니다.

- “conditional expression은 if문의 축약형일 뿐이다.”  
  → 정확히는 값을 반환하는 표현식입니다.

### Takeaway

conditional expression은 “간단한 값 선택”에만 사용합니다. 흐름 제어가 보이면 `if` 문으로 돌아갑니다.

## Item 8. Prevent Repetition with Assignment Expressions

### Central Question

값을 계산하면서 동시에 이름을 붙이는 것은 언제 도움이 될까?

### My Explanation

assignment expression, 즉 `:=`는 표현식 안에서 이름을 바인딩하고 그 값을 반환합니다.

```python
if (length := len("Python")) > 5:
    assert length == 6
```

핵심은 “한 번 계산한 값을 조건에서도 쓰고, 본문에서도 쓴다”는 상황입니다.

### Good Use: Avoid Duplicate Work

```python
def describe(text: str) -> str:
    if (length := len(text)) == 0:
        return "empty"
    return f"{length} characters"

assert describe("") == "empty"
assert describe("Python") == "6 characters"
```

`len(text)`를 반복하지 않고, 계산 결과에 이름을 붙였습니다.

### Good Use: Loop Until Empty

```python
from io import StringIO

stream = StringIO("Python\nLanguage\n")
chunks = []

while chunk := stream.read(7):
    chunks.append(chunk)

assert chunks == ["Python\n", "Languag", "e\n"]
```

파일을 일정 크기씩 읽는 루프처럼, “값을 가져오고 그 값이 비어 있지 않으면 처리한다”는 패턴에 잘 맞습니다.

### Bad Use: Hidden Logic

```python
# Too dense
if (user := find_user(user_id)) and (profile := user.profile) and profile.is_public:
    show(profile)
```

이 코드는 짧지만, 실패 가능한 단계가 너무 많이 한 줄에 들어 있습니다. 중간 값 각각이 중요한 의미를 갖는다면 분리하는 편이 좋습니다.

```python
def show_public_profile(user_id):
    user = find_user(user_id)
    if not user:
        return

    profile = user.profile
    if profile and profile.is_public:
        show(profile)
```

### Parentheses Rule

assignment expression은 위치에 따라 괄호가 필요할 수 있습니다. `if`와 `while`의 조건에서는 괄호 없이도 가능하지만, 많은 상황에서 괄호를 명시하면 읽기가 쉬워집니다.

```python
if length := len("abc"):
    assert length == 3

# 개인적으로는 아래처럼 쓰는 편이 더 눈에 잘 들어옵니다.
if (length := len("abc")):
    assert length == 3
```

### Common Misunderstandings

- “중복을 줄이면 항상 좋다.”  
  → 중복 제거 때문에 흐름이 숨겨지면 오히려 나쁩니다.

- “`:=`는 고급스럽기 때문에 자주 쓰면 좋다.”  
  → 자주 쓸수록 코드가 낯설어질 수 있습니다. 읽기 이득이 분명할 때만 사용합니다.

- “assignment statement와 assignment expression은 같다.”  
  → statement는 문장이고, expression은 값을 반환합니다. 이 차이가 중요합니다.

### Takeaway

assignment expression은 반복 계산을 줄이고 중간 값에 이름을 붙이는 도구입니다. 하지만 복잡한 흐름을 한 줄에 숨기는 도구가 되어서는 안 됩니다.

## Item 9. Consider match for Destructuring in Flow Control; Avoid When if Statements Are Sufficient

### Central Question

`match`는 단순한 `switch` 문인가, 아니면 구조를 해체하는 제어 흐름인가?

### My Explanation

Python의 `match`는 다른 언어의 단순한 `switch`와 다릅니다. 핵심은 **값의 구조를 pattern으로 비교하고, 필요한 경우 내부 값을 이름에 바인딩한다**는 점입니다.

```python
def describe_event(event: tuple) -> str:
    match event:
        case ("click", x, y):
            return f"click at ({x}, {y})"
        case ("keypress", key):
            return f"key pressed: {key}"
        case ("close",):
            return "window closed"
        case _:
            return "unknown event"

assert describe_event(("click", 10, 20)) == "click at (10, 20)"
assert describe_event(("keypress", "Enter")) == "key pressed: Enter"
assert describe_event(("close",)) == "window closed"
```

여기서 `match`는 단순히 첫 번째 값만 비교하지 않습니다. tuple의 길이와 내부 구조를 함께 봅니다.

### When `if` Is Better

조건이 단순한 boolean 판단이라면 `if`가 더 낫습니다.

```python
# Better with if
if user.is_active and user.email_verified:
    send_welcome_message(user)
```

이런 코드를 `match`로 바꾸면 오히려 의도가 흐려집니다. `match`는 “조건”보다 “구조”가 중요할 때 빛납니다.

### Pattern Binding

`case ("click", x, y)`에서 `x`, `y`는 기존 변수와 비교하는 것이 아니라 새 이름으로 바인딩됩니다.

```python
point = (3, 4)

match point:
    case (x, y):
        assert x == 3
        assert y == 4
```

이 점은 매우 중요합니다. pattern 안의 평범한 이름은 값을 “비교”하는 것이 아니라 값을 “받는” 이름입니다.

### Guards

추가 조건이 필요하면 guard를 사용할 수 있습니다.

```python
def quadrant(point: tuple[int, int]) -> str:
    match point:
        case (x, y) if x > 0 and y > 0:
            return "I"
        case (x, y) if x < 0 and y > 0:
            return "II"
        case (0, 0):
            return "origin"
        case _:
            return "other"

assert quadrant((1, 1)) == "I"
assert quadrant((-1, 1)) == "II"
assert quadrant((0, 0)) == "origin"
```

pattern이 구조를 확인하고, guard가 의미 조건을 확인합니다.

### Common Misunderstandings

- “`match`는 Python의 switch다.”  
  → 일부 상황에서는 비슷하게 쓸 수 있지만, 본질은 structural pattern matching입니다.

- “case 안의 이름은 기존 변수와 비교된다.”  
  → 대부분의 plain name은 새로 바인딩됩니다. 이 점을 모르면 매우 위험합니다.

- “복잡한 if-elif는 전부 match로 바꾸면 좋다.”  
  → 구조 분해가 없다면 `if`가 더 직접적일 수 있습니다.

- “`_`는 특별한 값이다.”  
  → pattern matching에서 `_`는 wildcard로 쓰입니다. 아무 값이나 받아들이되 이름을 바인딩하지 않습니다.

### Takeaway

`match`는 복잡한 구조를 가진 데이터를 읽기 좋게 분해할 때 강력합니다. 단순 조건 분기라면 `if`가 더 Python답습니다.
