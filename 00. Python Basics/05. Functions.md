## Chapter Thesis

함수는 Python 프로그램의 가장 중요한 사고 단위입니다. 좋은 함수는 코드를 줄이는 장치가 아니라, **입력과 출력 사이의 의미 있는 계약**입니다.

이 장의 핵심 질문은 다음과 같습니다.

- 함수가 받은 객체를 바꾸는가?
- 실패를 어떻게 표현하는가?
- 인자 이름은 호출자의 실수를 줄이는가?
- 기본값은 안전한가?
- 함수의 metadata는 보존되는가?

## Item 30. Know That Function Arguments Can Be Mutated

### Central Question

함수에 값을 넘기면 복사되는가, 같은 객체를 공유하는가?

### My Explanation

Python 함수 호출은 객체에 대한 reference를 전달합니다. 이름은 새로 생기지만, mutable object는 같은 객체를 가리킬 수 있습니다.

```python
def add_tag(tags: list[str], tag: str) -> None:
    tags.append(tag)

labels = ["python"]
add_tag(labels, "3.14")

assert labels == ["python", "3.14"]
```

함수 내부에서 `tags.append`를 호출하면 호출자가 가진 list도 바뀝니다. 반대로 이름을 새 객체로 재바인딩하는 것은 호출자에게 영향을 주지 않습니다.

```python
def reassign(tags: list[str]) -> None:
    tags = ["new"]

labels = ["old"]
reassign(labels)

assert labels == ["old"]
```

### Takeaway

Python에서 중요한 구분은 “값 전달/참조 전달”이라는 단순 구호보다 **객체 mutation과 이름 rebinding의 차이**입니다.

## Item 31. Return Dedicated Result Objects Instead of Requiring Function Callers to Unpack More Than Three Variables

### Central Question

함수 반환값이 많아지면 왜 tuple보다 이름 있는 결과가 나을까?

### My Explanation

작은 tuple 반환은 자연스럽습니다.

```python
def split_name(full_name: str) -> tuple[str, str]:
    first, last = full_name.split(maxsplit=1)
    return first, last

first, last = split_name("Ada Lovelace")
assert first == "Ada"
```

하지만 반환값이 많아지면 호출자가 순서를 기억해야 합니다.

이럴 때는 dataclass처럼 이름 있는 결과가 좋습니다.

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Summary:
    minimum: int
    maximum: int
    average: float
    count: int


def summarize(values: list[int]) -> Summary:
    return Summary(min(values), max(values), sum(values) / len(values), len(values))

summary = summarize([10, 20, 30])
assert summary.average == 20
```

### Takeaway

반환값이 많아질수록 위치보다 이름이 중요합니다. 이름 있는 결과 객체는 호출자의 기억 부담을 줄입니다.

## Item 32. Prefer Raising Exceptions to Returning None

### Central Question

실패를 `None`으로 표현하면 어떤 문제가 생길까?

### My Explanation

`None`은 값이 없음을 표현하기에 좋지만, 실패 이유를 설명하지 못합니다.

```python
def parse_int_or_none(text: str) -> int | None:
    try:
        return int(text)
    except ValueError:
        return None
```

호출자는 `None`을 반드시 확인해야 합니다. 확인을 잊으면 오류가 더 늦게, 더 엉뚱한 곳에서 발생합니다.

실패가 예외적인 상황이라면 예외로 표현하는 편이 명확합니다.

```python
def parse_int(text: str) -> int:
    try:
        return int(text)
    except ValueError as exc:
        raise ValueError(f"invalid integer: {text!r}") from exc

try:
    parse_int("x")
except ValueError as error:
    assert "invalid integer" in str(error)
```

### Takeaway

`None`은 정상적인 “값 없음”에 적합하고, 실패는 예외로 표현하는 편이 호출자와 디버거에게 친절합니다.

## Item 33. Know How Closures Interact with Variable Scope and nonlocal

### Central Question

중첩 함수는 바깥 scope의 변수를 어떻게 기억하고 바꿀까?

### My Explanation

closure는 함수가 정의될 때 주변 lexical scope의 이름을 기억하는 구조입니다.

```python
def make_multiplier(factor: int):
    def multiply(value: int) -> int:
        return value * factor
    return multiply

double = make_multiplier(2)
assert double(10) == 20
```

바깥 scope의 변수를 읽는 것은 자연스럽습니다. 하지만 재바인딩하려면 `nonlocal`이 필요합니다.

```python
def make_counter():
    count = 0

    def increment() -> int:
        nonlocal count
        count += 1
        return count

    return increment

counter = make_counter()
assert counter() == 1
assert counter() == 2
```

### Takeaway

closure는 상태를 숨길 수 있는 강력한 도구입니다. 하지만 상태가 복잡해지면 class가 더 명확할 수 있습니다.

## Item 34. Reduce Visual Noise with Variable Positional Arguments

### Central Question

`*args`는 언제 함수 호출을 더 자연스럽게 만들까?

### My Explanation

`*args`는 여러 positional argument를 tuple로 받습니다.

```python
def join_words(*words: str, separator: str = " ") -> str:
    return separator.join(words)

assert join_words("Python", "is", "clear") == "Python is clear"
```

호출자가 list를 만들 필요 없이 자연스럽게 값을 나열할 수 있습니다. 하지만 인자의 의미가 서로 다르다면 명시적인 parameter가 낫습니다.

```python
# Poor: each position has hidden meaning.
def create_user(*args):
    ...

# Better
def create_user(name: str, email: str, age: int | None = None):
    ...
```

### Takeaway

`*args`는 같은 의미의 값이 여러 개 올 때 적합합니다. 서로 다른 의미의 값을 숨기는 데 사용하면 API가 약해집니다.

## Item 35. Provide Optional Behavior with Keyword Arguments

### Central Question

옵션은 왜 keyword argument로 표현하는 것이 좋을까?

### My Explanation

옵션은 호출자가 이름을 보고 의미를 알 수 있어야 합니다.

```python
def render_title(title: str, *, uppercase: bool = False, prefix: str = "") -> str:
    text = title.upper() if uppercase else title
    return f"{prefix}{text}"

assert render_title("python", uppercase=True, prefix="# ") == "# PYTHON"
```

`uppercase=True`는 호출부에서 의도가 보입니다. 반면 positional boolean은 읽기 어렵습니다.

```python
# What does True mean?
# render_title("python", True, "# ")
```

### Takeaway

선택적 동작은 이름 있는 keyword argument로 표현합니다. 호출부가 문서처럼 읽히게 만드는 것이 목표입니다.

## Item 36. Use None and Docstrings to Specify Dynamic Default Arguments

### Central Question

기본값으로 mutable object를 쓰면 왜 위험할까?

### My Explanation

Python의 default argument는 함수가 정의될 때 한 번 평가됩니다.

```python
def append_bad(value: int, bucket: list[int] = []) -> list[int]:
    bucket.append(value)
    return bucket

assert append_bad(1) == [1]
assert append_bad(2) == [1, 2]
```

두 번째 호출이 첫 번째 호출의 list를 공유했습니다. 이것은 대부분 의도한 동작이 아닙니다.

안전한 방식은 `None`을 sentinel로 사용하는 것입니다.

```python
def append_good(value: int, bucket: list[int] | None = None) -> list[int]:
    """Return a bucket containing value; create a new bucket when omitted."""
    if bucket is None:
        bucket = []
    bucket.append(value)
    return bucket

assert append_good(1) == [1]
assert append_good(2) == [2]
```

### Takeaway

default argument는 호출 때마다 만들어지지 않습니다. 동적 기본값이 필요하면 `None`을 사용하고 docstring으로 정책을 설명합니다.

## Item 37. Enforce Clarity with Keyword-Only and Positional-Only Arguments

### Central Question

인자를 어떤 방식으로 전달해야 하는지도 API 계약이 될 수 있을까?

### My Explanation

Python은 positional-only(`/`)와 keyword-only(`*`) parameter를 지원합니다.

```python
def resize(image, /, width: int, height: int, *, keep_ratio: bool = False):
    return image, width, height, keep_ratio

assert resize("img", 100, 200, keep_ratio=True) == ("img", 100, 200, True)
```

- `/` 앞의 인자는 positional-only입니다.
- `*` 뒤의 인자는 keyword-only입니다.

keyword-only는 boolean option에 특히 유용합니다.

```python
def connect(host: str, port: int, *, secure: bool = True):
    return f"{host}:{port} secure={secure}"

assert connect("localhost", 443, secure=True) == "localhost:443 secure=True"
```

### Takeaway

인자 전달 방식은 API의 일부입니다. 모호한 옵션은 keyword-only로 만들어 호출부의 의미를 드러냅니다.

## Item 38. Define Function Decorators with functools.wraps

### Central Question

decorator가 원본 함수의 정체성을 훼손하지 않게 하려면?

### My Explanation

decorator는 함수를 감싸 새 함수를 반환합니다. 이때 metadata를 보존하지 않으면 이름, docstring, annotation이 흐려집니다.

```python
from functools import wraps


def trace(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@trace
def add(a: int, b: int) -> int:
    """Add two integers."""
    return a + b

assert add.__name__ == "add"
assert add.__doc__ == "Add two integers."
assert add(1, 2) == 3
```

### Takeaway

decorator를 작성할 때는 거의 항상 `functools.wraps`를 사용합니다. 함수의 정체성도 API입니다.

## Item 39. Prefer functools.partial over lambda Expressions for Glue Functions

### Central Question

함수에 일부 인자를 미리 묶고 싶을 때 어떤 표현이 더 명확할까?

### My Explanation

`partial`은 callable에 일부 인자를 미리 고정한 새 callable을 만듭니다.

```python
from functools import partial


def power(base: int, exponent: int) -> int:
    return base ** exponent

square = partial(power, exponent=2)

assert square(5) == 25
```

`lambda`로도 가능하지만 의도가 덜 직접적일 수 있습니다.

```python
square = lambda base: power(base, 2)
assert square(5) == 25
```

단순한 adapter는 lambda도 괜찮습니다. 하지만 “이 함수의 일부 인자를 고정한다”는 의도가 핵심이면 `partial`이 더 설명적입니다.

### Takeaway

`partial`은 함수 조합의 의도를 이름 있는 표준 도구로 표현합니다. lambda는 짧은 익명 계산에 적합합니다.
