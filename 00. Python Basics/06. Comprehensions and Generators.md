> 기준 버전: **Python 3.14.6**  
> 구조 출처: Effective Python 3rd Edition 공식 공개 목차  

## Chapter Thesis

Python의 comprehension과 generator는 반복문을 짧게 쓰는 문법이 아닙니다. 핵심은 **데이터 흐름을 선언적으로 표현할지, 값을 필요할 때까지 미룰지**를 선택하는 것입니다.

- comprehension은 “이 입력을 이런 출력 collection으로 바꾼다”는 선언입니다.
- generator는 “값을 한 번에 만들지 않고, 필요할 때 하나씩 생산한다”는 약속입니다.
- 복잡한 comprehension은 loop보다 나쁘고, 무분별한 generator는 디버깅을 어렵게 만듭니다.

## Item 40. Use Comprehensions Instead of map and filter

### Central Question

단순 변환과 필터링은 왜 comprehension이 더 읽기 쉬운가?

### My Explanation

comprehension은 입력, 조건, 출력 형태를 한눈에 보여줍니다.

```python
numbers = [1, 2, 3, 4, 5]
squares_of_even = [n * n for n in numbers if n % 2 == 0]

assert squares_of_even == [4, 16]
```

`map`과 `filter`도 같은 작업을 할 수 있지만, lambda가 섞이면 읽기 순서가 멀어집니다.

```python
squares_of_even = list(map(lambda n: n * n, filter(lambda n: n % 2 == 0, numbers)))
assert squares_of_even == [4, 16]
```

함수 이름이 이미 의미를 잘 드러내는 경우에는 `map`도 괜찮습니다.

```python
words = ["Python", "Rust"]
assert list(map(str.lower, words)) == ["python", "rust"]
```

### Takeaway

간단한 변환/필터링은 comprehension이 기본 선택입니다. 이미 좋은 함수 이름이 있고 조합이 단순하다면 `map`도 읽기 좋을 수 있습니다.

## Item 41. Avoid More Than Two Control Subexpressions in Comprehensions

### Central Question

comprehension은 언제 loop보다 더 어려워질까?

### My Explanation

comprehension은 한 줄에 여러 제어 흐름을 담을 수 있습니다.

```python
matrix = [[1, 2, 3], [4, 5, 6]]
flat_even = [value for row in matrix for value in row if value % 2 == 0]

assert flat_even == [2, 4, 6]
```

이 정도는 읽을 수 있습니다. 하지만 조건과 nested loop가 더 늘어나면 독자는 실행 순서를 머릿속으로 해석해야 합니다.

```python
# 의도가 즉시 보이지 않는다.
result = [x * y for x in range(5) if x % 2 for y in range(5) if y > x]
```

이럴 때는 명시적인 loop가 낫습니다.

```python
result = []
for x in range(5):
    if x % 2 == 0:
        continue
    for y in range(5):
        if y > x:
            result.append(x * y)
```

### Takeaway

comprehension은 변환의 구조가 즉시 보일 때만 좋습니다. 제어 표현식이 두 개를 넘어가면 loop나 helper function으로 분리합니다.

## Item 42. Reduce Repetition in Comprehensions with Assignment Expressions

### Central Question

comprehension 안에서 같은 계산이 반복될 때 어떻게 이름을 붙일까?

### My Explanation

assignment expression은 comprehension 안에서도 중간 결과에 이름을 줄 수 있습니다.

```python
texts = ["  Python  ", "", " Rust "]
cleaned = [value for text in texts if (value := text.strip())]

assert cleaned == ["Python", "Rust"]
```

`text.strip()`을 조건과 결과에서 반복하지 않고 `value`라는 이름을 부여했습니다.

하지만 assignment expression이 comprehension을 어렵게 만들면 분리하는 편이 좋습니다.

```python
def normalize_non_empty(text: str) -> str | None:
    value = text.strip()
    return value or None

cleaned = []
for text in texts:
    value = normalize_non_empty(text)
    if value is not None:
        cleaned.append(value)
```

### Takeaway

`:=`는 중복 계산을 줄일 수 있지만, comprehension의 읽기 난도를 높이면 helper function이나 loop가 더 좋습니다.

## Item 43. Consider Generators Instead of Returning Lists

### Central Question

함수가 결과를 모두 만들어서 반환해야 할까, 하나씩 생산해야 할까?

### My Explanation

list를 반환하는 함수는 모든 결과를 메모리에 만들어 둡니다.

```python
def squares_list(limit: int) -> list[int]:
    return [n * n for n in range(limit)]

assert squares_list(4) == [0, 1, 4, 9]
```

generator는 값을 필요할 때 하나씩 생산합니다.

```python
def squares(limit: int):
    for n in range(limit):
        yield n * n

assert list(squares(4)) == [0, 1, 4, 9]
```

값의 개수가 크거나, consumer가 일부만 사용할 수 있다면 generator가 더 자연스럽습니다.

```python
from itertools import islice

first_three = list(islice(squares(10_000_000), 3))
assert first_three == [0, 1, 4]
```

### Takeaway

결과 전체가 필요한 API라면 list를 반환하고, 순차 처리나 큰 데이터라면 generator를 고려합니다.

## Item 44. Consider Generator Expressions for Large List Comprehensions

### Central Question

collection을 만들 필요 없이 한 번만 소비한다면 무엇을 써야 할까?

### My Explanation

list comprehension은 즉시 list를 만듭니다.

```python
numbers = range(1_000)
total = sum([n * n for n in numbers])
```

`sum`은 iterator를 받을 수 있으므로 generator expression이 더 직접적입니다.

```python
total = sum(n * n for n in range(1_000))
assert total == 332833500
```

다만 같은 값을 여러 번 순회해야 한다면 generator expression은 적합하지 않습니다.

```python
values = (n for n in range(3))
assert list(values) == [0, 1, 2]
assert list(values) == []
```

### Takeaway

한 번만 소비할 값은 generator expression으로 충분합니다. 여러 번 사용하거나 길이가 필요한 값은 list로 materialize합니다.

## Item 45. Compose Multiple Generators with yield from

### Central Question

하위 generator의 값을 다시 전달할 때 왜 `yield from`이 유용할까?

### My Explanation

`yield from`은 다른 iterable 또는 generator에서 나오는 값을 그대로 위임합니다.

```python
def letters():
    yield from "ab"
    yield from "cd"

assert list(letters()) == ["a", "b", "c", "d"]
```

중첩 구조를 평탄하게 만들 때도 좋습니다.

```python
def flatten(rows: list[list[int]]):
    for row in rows:
        yield from row

assert list(flatten([[1, 2], [3]])) == [1, 2, 3]
```

직접 for loop로 다시 yield할 수도 있지만, `yield from`은 “위임”이라는 의도를 드러냅니다.

### Takeaway

generator가 다른 generator나 iterable의 값을 그대로 전달한다면 `yield from`은 간결하고 의미가 분명합니다.

## Item 46. Pass Iterators into Generators as Arguments Instead of Calling the send Method

### Central Question

generator에 외부 값을 공급하는 가장 읽기 쉬운 방식은 무엇일까?

### My Explanation

generator에는 `send`라는 고급 API가 있습니다. 하지만 대부분의 데이터 흐름에서는 generator에 iterator를 인자로 넘기는 편이 더 명확합니다.

```python
def running_total(values):
    total = 0
    for value in values:
        total += value
        yield total

assert list(running_total([1, 2, 3])) == [1, 3, 6]
```

이 구조는 입력이 어디서 들어오는지 명확합니다. 반면 `send`는 제어 흐름이 호출자와 generator 사이에 분산됩니다.

`send`는 coroutine-like protocol을 직접 다룰 때는 의미가 있지만, 일반적인 데이터 변환에는 과한 도구입니다.

### Takeaway

데이터 입력은 가능하면 iterator 인자로 표현합니다. `send`는 generator protocol 자체를 설계할 때만 신중하게 사용합니다.

## Item 47. Manage Iterative State Transitions with a Class Instead of the Generator throw Method

### Central Question

반복 중 상태 전이가 복잡해지면 generator로 계속 밀어붙여도 될까?

### My Explanation

generator는 단순한 생산 흐름에는 훌륭합니다. 하지만 외부 이벤트, 오류 주입, 상태 전이가 많아지면 generator protocol이 복잡해집니다.

이럴 때는 class가 더 명확할 수 있습니다.

```python
class CounterMachine:
    def __init__(self) -> None:
        self.count = 0

    def step(self, amount: int = 1) -> int:
        self.count += amount
        return self.count

    def reset(self) -> None:
        self.count = 0

machine = CounterMachine()
assert machine.step() == 1
assert machine.step(4) == 5
machine.reset()
assert machine.step() == 1
```

상태와 동작이 함께 이름을 얻었고, reset 같은 명령도 명확해졌습니다.

### Takeaway

generator는 값의 흐름을 표현하는 데 적합합니다. 상태 기계가 보이기 시작하면 class로 승격하는 것이 Python을 더 잘 이해한 선택입니다.
