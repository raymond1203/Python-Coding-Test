## Chapter Thesis

견고한 Python 코드는 “예외가 나지 않는 코드”가 아니라 **예외가 났을 때 무엇이 보장되는지 설명할 수 있는 코드**입니다. 이 챕터는 `try`, `with`, `raise`, `traceback`, `assert`, `exec/eval`을 단순 문법이 아니라 프로그램의 실패 모델을 설계하는 도구로 다룹니다.

Python을 깊게 이해한다는 것은 성공 경로만 읽는 것이 아니라, 실패 경로·정리 경로·관찰 경로를 함께 읽는 것입니다.

## Item 80. Take Advantage of Each Block in `try`/ `except`/ `else`/ `finally`

### Central Question

`try` 문 안의 각 블록은 어떤 책임을 가져야 할까?

### My Explanation

`try` 문은 네 부분으로 나뉩니다.

| Block | 실행 조건 | 좋은 용도 |
|---|---|---|
| `try` | 먼저 실행 | 실패 가능성이 있는 최소 코드 |
| `except` | 지정한 예외 발생 | 복구, 변환, 로깅 |
| `else` | 예외 없이 성공 | 성공 후 처리 |
| `finally` | 성공/실패와 무관 | 정리, release |

```python
def parse_positive_int(text: str) -> int:
    try:
        value = int(text)
    except ValueError as error:
        raise ValueError(f"not an integer: {text!r}") from error
    else:
        if value <= 0:
            raise ValueError("value must be positive")
        return value
    finally:
        # 로깅/정리처럼 성공 여부와 무관한 일을 둔다.
        pass
```

`else`는 성공 경로를 `try` 바깥으로 밀어내어, `except`가 잡는 범위를 좁혀줍니다. `finally`는 return이나 exception이 있어도 실행되므로, 상태를 덮어쓰는 복잡한 제어 흐름은 피해야 합니다.

### Takeaway

각 블록의 역할을 분리하면 실패 경계가 선명해집니다. 특히 `try`는 짧게, `else`는 성공 경로, `finally`는 정리 경로로 유지합니다.

## Item 81. `assert` Internal Assumptions and `raise` Missed Expectations

### Central Question

언제 `assert`를 쓰고, 언제 `raise`를 써야 할까?

### My Explanation

`assert`는 “이 지점에 도달했다면 반드시 참이어야 하는 내부 불변식”을 표현합니다. 사용자 입력 검증, 외부 API 오류, 파일 없음 같은 정상적인 실패 가능성은 명시적으로 `raise`해야 합니다.

```python
def average(numbers: list[float]) -> float:
    if not numbers:
        raise ValueError("numbers must not be empty")

    total = sum(numbers)
    count = len(numbers)
    assert count > 0  # 위 조건 때문에 내부적으로 보장되어야 함
    return total / count
```

`assert`는 최적화 모드(`python -O`)에서 제거될 수 있습니다. 따라서 프로그램의 보안이나 데이터 검증을 `assert`에 의존하면 안 됩니다.

### Takeaway

`assert`는 개발자에게 보내는 내부 계약이고, `raise`는 호출자에게 보내는 실패 계약입니다.

## Item 82. Consider `contextlib` and `with` Statements for Reusable `try`/ `finally` Behavior

### Central Question

반복되는 자원 정리 코드를 어떻게 안전하게 재사용할까?

### My Explanation

`with` 문은 enter/exit 경계를 통해 자원 수명을 코드 구조에 고정합니다.

```python
from pathlib import Path

path = Path("data.txt")
with path.open("w", encoding="utf-8") as file:
    file.write("hello")
```

직접 context manager를 만들 때는 `contextlib.contextmanager`가 유용합니다.

```python
from contextlib import contextmanager
from time import perf_counter
from collections.abc import Iterator

@contextmanager
def timer(label: str) -> Iterator[None]:
    start = perf_counter()
    try:
        yield
    finally:
        elapsed = perf_counter() - start
        print(f"{label}: {elapsed:.3f}s")

with timer("work"):
    sum(range(100_000))
```

이 구조는 성공/실패와 무관하게 정리 코드가 실행되도록 강제합니다.

### Takeaway

`with`는 단순 문법 설탕이 아니라 자원 수명을 표현하는 설계 도구입니다. 반복되는 `try/finally`는 context manager로 승격합니다.

## Item 83. Always Make `try` Blocks as Short as Possible

### Central Question

왜 큰 `try` 블록은 위험할까?

### My Explanation

`try` 블록이 길면 `except`가 어떤 코드의 예외를 처리하는지 흐려집니다.

```python
# 좋지 않음: int 변환 오류와 다른 버그가 섞일 수 있음
try:
    value = int(user_input)
    normalized = 100 / value
    result = expensive_lookup(normalized)
except ValueError:
    result = None
```

더 좋은 구조는 실패 가능성이 같은 코드만 묶는 것입니다.

```python
try:
    value = int(user_input)
except ValueError:
    result = None
else:
    normalized = 100 / value
    result = expensive_lookup(normalized)
```

이렇게 하면 `expensive_lookup` 내부의 `ValueError`를 실수로 삼키지 않습니다.

### Takeaway

짧은 `try`는 예외 처리의 의미를 명확하게 만듭니다. 넓은 `try`는 버그를 정상적인 실패처럼 보이게 만들 수 있습니다.

## Item 84. Beware of Exception Variables Disappearing

### Central Question

`except Exception as error`의 `error` 변수는 어디까지 살아 있을까?

### My Explanation

Python은 예외 객체가 traceback을 통해 지역 변수들을 참조하면서 reference cycle을 만들 수 있기 때문에, `except ... as name`의 `name`을 블록 이후에 정리합니다.

```python
try:
    raise ValueError("bad")
except ValueError as error:
    message = str(error)

# 여기서 error를 다시 쓰려고 하면 NameError가 날 수 있다.
assert message == "bad"
```

예외 정보를 블록 밖에서 써야 한다면 필요한 값만 별도 변수에 저장합니다.

### Takeaway

예외 객체 자체를 오래 들고 있기보다 필요한 메시지, 타입, traceback 정보를 의도적으로 추출합니다.

## Item 85. Beware of Catching the `Exception` Class

### Central Question

`except Exception:`은 왜 조심해야 할까?

### My Explanation

`Exception`은 대부분의 일반 예외를 잡기 때문에 복구할 수 없는 버그까지 숨길 수 있습니다.

```python
try:
    process_record(record)
except Exception as error:
    logger.exception("failed to process record")
    raise  # 복구할 수 없다면 다시 올린다
```

넓은 포착이 필요한 경우도 있습니다. 예를 들어 worker loop에서 한 작업의 실패가 전체 서비스를 죽이지 않게 할 수 있습니다. 하지만 이 경우에도 로깅, 격리, 재시도 정책, 최종 실패 전달이 있어야 합니다.

좋은 원칙은 다음과 같습니다.

- 가능하면 구체적인 예외를 잡는다.
- 복구할 수 없는 예외는 다시 `raise`한다.
- logging 없이 삼키지 않는다.
- 테스트에서 예외가 숨지 않는지 확인한다.

### Takeaway

넓은 예외 포착은 시스템 경계에서만 제한적으로 사용합니다. 내부 로직에서는 구체적 예외를 선호합니다.

## Item 86. Understand the Difference Between `Exception` and `BaseException`

### Central Question

왜 `BaseException`을 거의 잡지 말아야 할까?

### My Explanation

`BaseException`은 모든 예외의 최상위 계층이며, `KeyboardInterrupt`, `SystemExit`, `GeneratorExit` 같은 종료 신호도 포함합니다. 일반 애플리케이션 로직에서 이것들을 삼키면 프로그램이 정상적으로 종료되지 않을 수 있습니다.

```python
try:
    run_forever()
except KeyboardInterrupt:
    print("shutdown requested")
```

종료 신호를 다루려면 명시적으로 다루고, 일반 오류 처리는 `Exception` 아래의 구체적 예외를 잡는 편이 안전합니다.

### Takeaway

`BaseException`은 프로그램 제어 신호까지 포함합니다. 일반 오류 복구에는 사용하지 않는 것이 원칙입니다.

## Item 87. Use `traceback` for Enhanced Exception Reporting

### Central Question

예외를 단순 문자열이 아니라 진단 가능한 정보로 남기려면?

### My Explanation

`traceback` 모듈은 예외 타입, 메시지, 호출 스택을 구조화된 문자열로 만들 수 있습니다.

```python
import traceback

try:
    1 / 0
except ZeroDivisionError:
    report = traceback.format_exc()
    print(report)
```

실무에서는 `logging.exception()`을 함께 쓰는 경우가 많습니다. 이 함수는 현재 예외의 traceback을 로그에 포함합니다.

```python
import logging

try:
    1 / 0
except ZeroDivisionError:
    logging.exception("calculation failed")
```

### Takeaway

견고성은 실패를 숨기는 것이 아니라 실패를 재현 가능하게 남기는 것입니다. traceback은 그 출발점입니다.

## Item 88. Consider Explicitly Chaining Exceptions to Clarify Tracebacks

### Central Question

추상화 경계에서 예외를 바꿀 때 원인을 어떻게 보존할까?

### My Explanation

낮은 수준의 예외를 높은 수준의 도메인 예외로 바꾸고 싶을 때 `raise NewError(...) from error`를 사용하면 원인 관계가 traceback에 남습니다.

```python
class ConfigError(Exception):
    pass


def read_port(text: str) -> int:
    try:
        return int(text)
    except ValueError as error:
        raise ConfigError(f"invalid port: {text!r}") from error
```

`from None`을 사용하면 내부 원인을 의도적으로 숨길 수 있지만, 디버깅 정보가 줄어드므로 API 경계에서 신중히 사용해야 합니다.

### Takeaway

예외 변환은 좋은 추상화입니다. 단, 원인까지 보존해야 좋은 디버깅 경험이 됩니다.

## Item 89. Always Pass Resources into Generators and Have Callers Clean Them Up Outside

### Central Question

generator 내부에서 파일 같은 자원을 열면 어떤 문제가 생길까?

### My Explanation

generator는 lazy evaluation을 합니다. 따라서 generator 내부에서 자원을 열면, 소비가 중간에 멈췄을 때 정리 시점이 모호해질 수 있습니다.

```python
# 피하고 싶은 구조
# def read_lines(path):
#     with open(path, encoding="utf-8") as file:
#         for line in file:
#             yield line.strip()
```

더 명확한 구조는 호출자가 자원 수명을 관리하고, generator는 이미 열린 자원이나 iterable을 받는 것입니다.

```python
from collections.abc import Iterable, Iterator


def normalized_lines(lines: Iterable[str]) -> Iterator[str]:
    for line in lines:
        text = line.strip()
        if text:
            yield text

with open("data.txt", encoding="utf-8") as file:
    for line in normalized_lines(file):
        print(line)
```

### Takeaway

generator는 데이터 변환에 집중시키고, 자원 수명은 호출자가 `with`로 관리하게 하면 정리 경계가 선명해집니다.

## Item 90. Never Set `__debug__` to `False`

### Central Question

`__debug__`는 직접 바꿀 수 있는 설정값일까?

### My Explanation

`__debug__`는 Python 실행 모드에 의해 정해지는 내장 상수입니다. 일반 모드에서는 `True`, 최적화 모드(`-O`)에서는 `False`가 됩니다. 사용자가 코드에서 직접 대입하려고 하면 문법 오류가 납니다.

```python
if __debug__:
    print("debug mode")
```

중요한 점은 `assert` 문이 `__debug__`와 연결되어 있다는 것입니다. 최적화 모드에서는 assert 검사가 제거될 수 있으므로, 필수 검증을 assert로 작성하면 안 됩니다.

### Takeaway

`__debug__`는 실행 환경의 신호이지 애플리케이션 설정이 아닙니다. 필수 검증은 항상 명시적인 `if`와 `raise`로 작성합니다.

## Item 91. Avoid `exec` and `eval` Unless You’re Building a Developer Tool

### Central Question

동적으로 코드를 실행하는 기능은 왜 위험할까?

### My Explanation

`eval`은 표현식을 평가하고, `exec`는 문장을 실행합니다. 둘 다 문자열을 코드로 바꾸기 때문에 입력이 외부에서 온다면 보안 문제가 됩니다.

```python
# 위험한 예: 사용자 입력을 코드로 실행
# result = eval(user_input)
```

대부분의 경우 더 안전한 대안이 있습니다.

| 목표 | 대안 |
|---|---|
| JSON 데이터 읽기 | `json.loads` |
| Python literal 읽기 | `ast.literal_eval` |
| 이름으로 함수 선택 | dict dispatch |
| 템플릿 렌더링 | 전용 템플릿 엔진 |

```python
operations = {
    "add": lambda a, b: a + b,
    "mul": lambda a, b: a * b,
}

operation = operations[user_choice]
print(operation(2, 3))
```

### Takeaway

`exec`와 `eval`은 개발자 도구, REPL, DSL 구현처럼 코드 실행 자체가 목적일 때만 신중히 사용합니다. 일반 애플리케이션 로직에서는 데이터 구조와 명시적 dispatch를 선호합니다.

## Chapter Summary

| Robustness Principle | Practice |
|---|---|
| 실패 경계 축소 | `try` 블록을 짧게 유지한다 |
| 복구 가능성 명시 | 구체적 예외를 잡고, 필요하면 다시 올린다 |
| 자원 수명 보장 | `with`, `contextlib`, `finally`를 사용한다 |
| 원인 보존 | `raise ... from error`로 예외를 연결한다 |
| 위험 기능 격리 | `exec`, `eval`, 넓은 `Exception` 포착을 제한한다 |
