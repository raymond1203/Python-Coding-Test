## Chapter Thesis

테스트는 “정답 확인”이 아니라 **코드의 계약을 실행 가능한 문서로 남기는 일**입니다. 디버깅은 “감으로 고치는 일”이 아니라 **실패를 재현하고 관찰해서 원인을 좁히는 과정**입니다.

Python을 깊게 이해하는 사람은 `unittest`, `mock`, `pdb`, `tracemalloc`을 단순 도구로 쓰지 않습니다. 테스트 격리, 의존성 경계, 수치 오차, 메모리 사용량까지 프로그램의 신뢰성 모델로 다룹니다.

## Item 108. Verify Related Behaviors in `TestCase` Subclasses

### Central Question

테스트 코드는 어떻게 구조화해야 읽기 쉬울까?

### My Explanation

`unittest.TestCase`는 관련된 행동을 한 클래스에 묶고, 각 테스트 메서드가 독립된 사례를 검증하게 합니다.

```python
import unittest


def normalize_name(name: str) -> str:
    return " ".join(name.strip().split()).title()


class NormalizeNameTest(unittest.TestCase):
    def test_strips_outer_spaces(self) -> None:
        self.assertEqual(normalize_name("  ada  "), "Ada")

    def test_collapses_inner_spaces(self) -> None:
        self.assertEqual(normalize_name("ada   lovelace"), "Ada Lovelace")

if __name__ == "__main__":
    unittest.main()
```

좋은 테스트 이름은 실패했을 때 무엇이 깨졌는지 설명합니다. `test_1`, `test_normal`보다 `test_collapses_inner_spaces`가 더 좋은 문서입니다.

### Takeaway

테스트 클래스는 코드 구조가 아니라 행동 단위를 기준으로 묶습니다. 테스트 이름은 실패 메시지의 첫 문장입니다.

## Item 109. Prefer Integration Tests over Unit Tests

### Central Question

unit test가 많으면 충분할까?

### My Explanation

unit test는 빠르고 정확히 실패 위치를 알려주지만, 실제 시스템 경계가 연결될 때 생기는 문제를 놓칠 수 있습니다. integration test는 여러 구성요소를 실제 흐름에 가깝게 실행해 계약이 맞는지 확인합니다.

예를 들어 parser 함수만 테스트하는 것도 필요하지만, 파일을 읽고 파싱하고 결과를 저장하는 흐름도 테스트해야 합니다.

```python
from pathlib import Path


def load_numbers(path: Path) -> list[int]:
    return [int(line) for line in path.read_text().splitlines() if line]


def test_load_numbers_integration(tmp_path: Path) -> None:
    path = tmp_path / "numbers.txt"
    path.write_text("1\n2\n3\n", encoding="utf-8")
    assert load_numbers(path) == [1, 2, 3]
```

위 예시는 `pytest` 스타일 임시 경로를 사용하지만, 핵심은 framework가 아니라 실제 파일 I/O 경계를 포함한다는 점입니다.

### Takeaway

unit test는 부품을 확인하고, integration test는 조립 상태를 확인합니다. 포트폴리오 품질의 코드는 둘 다 필요하지만, 사용자 가치에 가까운 integration path를 반드시 포함해야 합니다.

## Item 110. Isolate Tests From Each Other with `setUp`, `tearDown`, `setUpModule`, and `tearDownModule`

### Central Question

테스트가 실행 순서에 따라 성공/실패하면 무엇이 문제일까?

### My Explanation

테스트는 서로 독립이어야 합니다. `setUp`은 각 테스트 전에 새 상태를 만들고, `tearDown`은 테스트 후 정리를 담당합니다.

```python
import unittest

class StackTest(unittest.TestCase):
    def setUp(self) -> None:
        self.stack: list[int] = []

    def test_push(self) -> None:
        self.stack.append(1)
        self.assertEqual(self.stack, [1])

    def test_empty_by_default(self) -> None:
        self.assertEqual(self.stack, [])
```

외부 리소스가 모듈 전체에서 한 번만 필요하다면 `setUpModule`과 `tearDownModule`을 사용할 수 있지만, 공유 상태가 테스트 간 오염을 만들지 않게 주의해야 합니다.

### Takeaway

좋은 테스트는 순서와 반복 실행에 강합니다. 각 테스트는 자신에게 필요한 상태를 직접 만들고 직접 정리해야 합니다.

## Item 111. Use Mocks to Test Code with Complex Dependencies

### Central Question

네트워크, 시간, 외부 API에 의존하는 코드는 어떻게 테스트할까?

### My Explanation

`unittest.mock`은 복잡하거나 느리거나 불안정한 의존성을 테스트 안에서 통제하게 해줍니다.

```python
from unittest.mock import Mock


def send_welcome(email_client, address: str) -> bool:
    response = email_client.send(address, "Welcome")
    return response == "ok"


def test_send_welcome() -> None:
    client = Mock()
    client.send.return_value = "ok"

    assert send_welcome(client, "ada@example.com") is True
    client.send.assert_called_once_with("ada@example.com", "Welcome")
```

mock은 강력하지만, 구현 세부사항에 과도하게 묶이면 리팩터링에 취약해집니다. mock의 목표는 “내 코드가 외부 계약을 올바르게 사용한다”를 검증하는 것입니다.

### Takeaway

mock은 외부 세계를 통제하는 도구입니다. 하지만 테스트가 실제 행동보다 내부 호출 방식에 집착하지 않도록 경계해야 합니다.

## Item 112. Encapsulate Dependencies to Facilitate Mocking and Testing

### Central Question

테스트하기 쉬운 코드는 어떤 구조를 가질까?

### My Explanation

의존성을 함수 내부에서 직접 생성하면 테스트에서 바꾸기 어렵습니다.

```python
# 테스트하기 어려운 구조
# def notify(address: str) -> None:
#     client = RealEmailClient()
#     client.send(address, "hello")
```

대신 의존성을 밖에서 주입하거나, 작은 interface로 감싸면 테스트가 쉬워집니다.

```python
from typing import Protocol

class EmailClient(Protocol):
    def send(self, address: str, message: str) -> str: ...


def notify(client: EmailClient, address: str) -> bool:
    return client.send(address, "hello") == "ok"
```

이 구조에서는 실제 client, fake client, mock client를 모두 같은 함수에 넣을 수 있습니다.

### Takeaway

테스트 가능성은 나중에 붙이는 장식이 아니라 설계의 결과입니다. 외부 의존성은 생성과 사용을 분리해야 합니다.

## Item 113. Use `assertAlmostEqual` to Control Precision in Floating Point Tests

### Central Question

부동소수 결과는 어떻게 테스트해야 할까?

### My Explanation

이진 부동소수점에서는 작은 오차가 자연스럽습니다. 따라서 정확한 동등성보다 허용 오차를 테스트 계약으로 명시해야 합니다.

```python
import unittest

class FloatTest(unittest.TestCase):
    def test_sum(self) -> None:
        self.assertAlmostEqual(0.1 + 0.2, 0.3, places=7)
```

허용 오차는 도메인 요구사항입니다. 금융처럼 십진 정밀도가 중요한 영역에서는 `Decimal`을 사용하고, 물리/통계 계산에서는 상대/절대 오차 기준을 명시합니다.

### Takeaway

floating point 테스트는 “같다/다르다”가 아니라 “허용 가능한 오차 안에 있다”를 검증합니다.

## Item 114. Consider Interactive Debugging with `pdb`

### Central Question

로그만으로 원인을 찾기 어려울 때 어떻게 실행 상태를 볼까?

### My Explanation

`pdb`는 Python 표준 interactive debugger입니다. 특정 지점에서 실행을 멈추고 변수, call stack, 다음 실행 흐름을 확인할 수 있습니다.

```python
def divide(a: int, b: int) -> float:
    breakpoint()  # Python 3.7+; 기본적으로 pdb 진입
    return a / b
```

자주 쓰는 명령은 다음과 같습니다.

| Command | Meaning |
|---|---|
| `n` | next line |
| `s` | step into |
| `c` | continue |
| `p expr` | expression 출력 |
| `bt` | stack trace |

테스트 실패 지점에서 debugger를 쓰면 입력 값과 중간 상태를 직접 확인할 수 있습니다.

### Takeaway

디버거는 추측을 줄이는 도구입니다. 단, 디버깅 후 남은 `breakpoint()`가 commit되지 않도록 주의합니다.

## Item 115. Use `tracemalloc` to Understand Memory Usage and Leaks

### Central Question

메모리가 계속 늘어날 때 어디서 할당되는지 어떻게 찾을까?

### My Explanation

`tracemalloc`은 Python 메모리 할당을 추적하고 snapshot 간 차이를 비교할 수 있습니다.

```python
import tracemalloc

tracemalloc.start()

before = tracemalloc.take_snapshot()
leaky = [bytearray(1024) for _ in range(1000)]
after = tracemalloc.take_snapshot()

for stat in after.compare_to(before, "lineno")[:5]:
    print(stat)
```

메모리 문제를 볼 때는 다음을 구분합니다.

- 진짜 leak인가, cache인가?
- Python 객체인가, native extension 메모리인가?
- 특정 요청 이후에만 증가하는가?
- snapshot 비교에서 같은 line이 반복되는가?

### Takeaway

메모리 디버깅도 측정과 비교의 문제입니다. `tracemalloc`은 “어디서 늘어났는가?”라는 질문에 답하게 해줍니다.
