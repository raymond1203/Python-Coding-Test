## Chapter Thesis

`dict`는 Python에서 단순한 자료구조가 아니라, 이름과 값을 연결하는 핵심 추상화입니다.

Python의 module namespace, object attribute, JSON-like data, cache, index, frequency table은 모두 mapping 사고와 연결됩니다.

이 장의 목표는 dictionary API를 외우는 것이 아니라 다음을 이해하는 것입니다.

- key는 hashable해야 한다.
- missing key는 설계 문제다.
- insertion order는 편리하지만 의미적 의존에는 신중해야 한다.
- nested dict가 깊어지면 데이터 모델을 다시 봐야 한다.

## Item 25. Be Cautious when Relying on Dictionary Insertion Ordering

### Central Question

`dict`가 순서를 보존한다는 사실은 어떤 의미까지 보장할까?

### My Explanation

현대 Python의 `dict`는 insertion order를 보존합니다.

```python
profile = {}
profile["name"] = "Ada"
profile["language"] = "Python"
profile["year"] = 1991

assert list(profile) == ["name", "language", "year"]
```

이것은 매우 유용합니다. 하지만 “삽입 순서가 보존된다”와 “이 데이터의 의미가 순서에 의존한다”는 다릅니다.

정말 순서가 핵심인 데이터라면 list of tuples, dataclass list, explicit order field처럼 순서를 모델링하는 편이 더 명확할 수 있습니다.

```python
steps = [
    (1, "parse"),
    (2, "validate"),
    (3, "save"),
]
```

### Takeaway

`dict`의 순서 보존은 편리한 특성입니다. 그러나 비즈니스 의미가 순서에 있다면 순서를 더 명시적인 데이터 모델로 표현합니다.

## Item 26. Prefer get over in and KeyError to Handle Missing Dictionary Keys

### Central Question

key가 없을 때의 정책은 어디에 드러나야 할까?

### My Explanation

값이 없을 때 기본값을 쓰는 것은 흔한 작업입니다.

```python
counts = {"python": 3}

assert counts.get("java", 0) == 0
```

`get`은 “없으면 이 값을 사용한다”는 정책을 한 곳에 보여줍니다.

```python
word = "python"
counts[word] = counts.get(word, 0) + 1

assert counts["python"] == 4
```

`in`으로 먼저 검사할 수도 있지만, 같은 key 조회가 반복됩니다.

```python
if word in counts:
    counts[word] += 1
else:
    counts[word] = 1
```

`KeyError`를 사용하는 방식은 missing이 예외적인 상황일 때 적합합니다. 정상 흐름의 일부라면 `get`, `defaultdict`, `setdefault` 등을 고려합니다.

### Takeaway

missing key가 정상 흐름이면 `get` 같은 API로 정책을 드러내고, 정말 비정상 상황이면 `KeyError`를 그대로 두거나 명시적으로 처리합니다.

## Item 27. Prefer defaultdict over setdefault to Handle Missing Items in Internal State

### Central Question

기본값 생성이 반복된다면 누가 그 책임을 가져야 할까?

### My Explanation

`defaultdict`는 key가 없을 때 기본값을 자동으로 만듭니다.

```python
from collections import defaultdict

groups: defaultdict[str, list[str]] = defaultdict(list)

groups["language"].append("Python")
groups["language"].append("Rust")

assert groups["language"] == ["Python", "Rust"]
```

`setdefault`도 비슷한 일을 할 수 있습니다.

```python
groups = {}
groups.setdefault("language", []).append("Python")
```

하지만 내부 상태로 계속 누적하는 구조라면 `defaultdict`가 의도를 더 잘 드러냅니다.

- 이 mapping은 missing key에 대해 list를 만든다.
- 각 key는 여러 값을 누적한다.
- 기본값 정책이 dictionary 생성 시점에 고정된다.

`defaultdict`는 읽기만 해도 key가 생길 수 있다는 점은 주의해야 합니다.

```python
groups = defaultdict(list)
_ = groups["missing"]

assert "missing" in groups
```

외부 API로 노출되는 dictionary라면 일반 dict로 변환하는 것도 좋습니다.

```python
public_groups = dict(groups)
```

### Takeaway

기본값 생성이 자료구조의 본질이면 `defaultdict`를 사용합니다. 단, key 생성 부작용은 이해하고 사용해야 합니다.

## Item 28. Know How to Construct Key-Dependent Default Values with __missing__

### Central Question

기본값이 key 자체에 의존한다면 어떻게 표현할 수 있을까?

### My Explanation

`defaultdict`의 factory는 key를 받지 않습니다. 기본값이 key에 따라 달라져야 한다면 `dict.__missing__`을 구현할 수 있습니다.

```python
class SlugMap(dict):
    def __missing__(self, key: str) -> str:
        value = key.strip().lower().replace(" ", "-")
        self[key] = value
        return value

slugs = SlugMap()

assert slugs["Hello Python"] == "hello-python"
assert slugs["Hello Python"] == "hello-python"
```

`__missing__`은 `dict[key]` 접근에서 key가 없을 때 호출됩니다. `get` 같은 메서드와의 관계는 다르게 느껴질 수 있으므로 API 설계에 주의해야 합니다.

```python
slugs = SlugMap()
assert slugs.get("Hello Python") is None
assert slugs["Hello Python"] == "hello-python"
```

### Takeaway

key-dependent default가 필요하면 `__missing__`이 강력합니다. 하지만 일반적인 코드에서는 `get`, `defaultdict`가 더 단순하고 읽기 쉽습니다.

## Item 29. Compose Classes Instead of Deeply Nesting Dictionaries, Lists, and Tuples

### Central Question

중첩 dictionary는 언제 클래스로 승격되어야 할까?

### My Explanation

초기에는 nested dict가 편합니다.

```python
student = {
    "name": "Ada",
    "scores": {"math": 95, "python": 100},
}
```

하지만 구조가 깊어질수록 문제가 생깁니다.

- key 이름을 틀려도 런타임까지 모를 수 있다.
- 어떤 field가 필수인지 드러나지 않는다.
- 내부 구조를 모든 사용자가 알아야 한다.
- refactoring이 어렵다.

이럴 때는 이름 있는 타입을 도입합니다.

```python
from dataclasses import dataclass, field

@dataclass
class Student:
    name: str
    scores: dict[str, int] = field(default_factory=dict)

    def average(self) -> float:
        if not self.scores:
            return 0.0
        return sum(self.scores.values()) / len(self.scores)

student = Student("Ada", {"math": 95, "python": 100})
assert student.average() == 97.5
```

클래스는 데이터를 숨기는 도구가 아니라, **데이터의 의미와 동작을 함께 묶는 도구**입니다.

### Takeaway

중첩 구조가 설명을 필요로 하기 시작하면, 그 설명을 class나 dataclass로 옮길 시점입니다.
