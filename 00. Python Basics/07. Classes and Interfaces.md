## Chapter Thesis

Python의 class는 Java식 class hierarchy를 강제하기 위한 도구가 아닙니다. Python에서 class는 주로 **상태와 동작에 이름을 붙이고, 객체가 어떤 protocol을 만족하는지 드러내는 도구**입니다.

이 장의 핵심은 “상속을 어떻게 쓰는가”보다 “객체가 어떤 역할을 하는가”입니다.

## Item 48. Accept Functions Instead of Classes for Simple Interfaces

### Central Question

작은 사용자 정의 동작을 위해 꼭 class가 필요할까?

### My Explanation

Python에서는 함수도 first-class object입니다. 단일 동작만 필요한 interface라면 class보다 함수가 더 단순합니다.

```python
def sort_key(user: dict) -> tuple[int, str]:
    return (-user["score"], user["name"])

users = [
    {"name": "Ada", "score": 90},
    {"name": "Guido", "score": 95},
]

users.sort(key=sort_key)
assert users[0]["name"] == "Guido"
```

동작이 하나라면 class로 감싸기보다 callable을 받는 API가 더 가볍습니다.

### Takeaway

interface가 “하나의 행동”이면 함수가 자연스럽습니다. 상태와 여러 동작이 함께 필요해질 때 class를 고려합니다.

## Item 49. Prefer Object-Oriented Polymorphism over Functions with isinstance Checks

### Central Question

타입별 분기를 함수에 모을까, 객체가 스스로 행동하게 할까?

### My Explanation

`isinstance`가 반복되면 동작이 객체 밖으로 새고 있다는 신호일 수 있습니다.

```python
class Shape:
    def area(self) -> float:
        raise NotImplementedError

class Rectangle(Shape):
    def __init__(self, width: float, height: float) -> None:
        self.width = width
        self.height = height

    def area(self) -> float:
        return self.width * self.height

class Circle(Shape):
    def __init__(self, radius: float) -> None:
        self.radius = radius

    def area(self) -> float:
        return 3.14159 * self.radius * self.radius

shapes: list[Shape] = [Rectangle(3, 4), Circle(2)]
areas = [shape.area() for shape in shapes]

assert areas[0] == 12
```

다형성의 장점은 새로운 타입이 추가될 때 기존 분기 함수를 계속 수정하지 않아도 된다는 점입니다.

### Takeaway

타입에 따라 행동이 달라진다면, 행동을 객체 안으로 옮기는 것이 더 안정적인 설계일 수 있습니다.

## Item 50. Consider functools.singledispatch for Functional-Style Programming Instead of Object-Oriented Polymorphism

### Central Question

객체에 메서드를 추가할 수 없을 때 타입별 처리는 어떻게 정리할까?

### My Explanation

`singledispatch`는 첫 번째 인자의 타입에 따라 다른 함수를 호출하게 합니다.

```python
from functools import singledispatch

@singledispatch
def describe(value) -> str:
    return f"object:{value!r}"

@describe.register
def _(value: int) -> str:
    return f"int:{value}"

@describe.register
def _(value: list) -> str:
    return f"list[{len(value)}]"

assert describe(3) == "int:3"
assert describe([1, 2]) == "list[2]"
```

이 방식은 외부 타입을 다루거나 class hierarchy를 수정할 수 없을 때 유용합니다. 하지만 dispatch 기준이 첫 번째 인자 중심이라는 한계도 있습니다.

### Takeaway

객체의 method로 표현하기 어려운 타입별 동작은 `singledispatch`로 정리할 수 있습니다. 다만 핵심 domain 객체에는 다형성이 더 자연스러울 수 있습니다.

## Item 51. Prefer dataclasses for Defining Lightweight Classes

### Central Question

데이터를 담는 class를 직접 boilerplate로 작성해야 할까?

### My Explanation

`dataclass`는 데이터 중심 class의 반복 코드를 줄입니다.

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

point = Point(3, 4)
assert point.x == 3
assert repr(point) == "Point(x=3, y=4)"
```

직접 `__init__`, `__repr__`, `__eq__`를 작성하지 않아도 의미 있는 기본 동작이 생깁니다.

### Takeaway

값을 담는 작은 타입에는 dataclass를 먼저 고려합니다. class가 행동 중심으로 커지면 method를 추가해 의미를 강화합니다.

## Item 52. Use @classmethod Polymorphism to Construct Objects Generically

### Central Question

생성 로직을 instance 밖 어디에 두어야 할까?

### My Explanation

`@classmethod`는 class 자체를 첫 번째 인자로 받습니다. 하위 class에서도 같은 생성 API를 유지할 수 있습니다.

```python
from dataclasses import dataclass

@dataclass
class User:
    name: str
    email: str

    @classmethod
    def from_record(cls, record: dict[str, str]):
        return cls(name=record["name"], email=record["email"])

user = User.from_record({"name": "Ada", "email": "ada@example.com"})
assert user.name == "Ada"
```

`cls(...)`를 사용하면 subclass가 호출해도 subclass instance를 만들 수 있습니다.

### Takeaway

문자열, dict, row 같은 외부 표현에서 객체를 만드는 로직은 `@classmethod`로 이름 붙일 수 있습니다.

## Item 53. Initialize Parent Classes with super

### Central Question

상속 구조에서 부모 초기화는 어떻게 협력적으로 처리할까?

### My Explanation

`super()`는 현재 class 다음의 method resolution order를 따라 method를 호출합니다.

```python
class Timestamped:
    def __init__(self, created_at: str) -> None:
        self.created_at = created_at

class Document(Timestamped):
    def __init__(self, title: str, created_at: str) -> None:
        super().__init__(created_at)
        self.title = title

doc = Document("Notes", "2026-07-02")
assert doc.created_at == "2026-07-02"
```

직접 부모 class 이름을 호출하면 상속 구조가 바뀔 때 깨지기 쉽습니다. 다중 상속에서는 특히 `super()`의 협력적 호출 모델이 중요합니다.

### Takeaway

상속을 사용한다면 `super()`를 기본으로 생각합니다. 하지만 상속 자체가 필요한지도 먼저 검토합니다.

## Item 54. Consider Composing Functionality with Mix-in Classes

### Central Question

공통 기능을 상속으로 섞어 넣는 것은 언제 좋은가?

### My Explanation

mix-in은 독립적인 작은 기능을 여러 class에 조합하기 위한 class입니다.

```python
class AsDictMixin:
    def as_dict(self) -> dict:
        return dict(self.__dict__)

class Product(AsDictMixin):
    def __init__(self, name: str, price: int) -> None:
        self.name = name
        self.price = price

product = Product("Book", 30000)
assert product.as_dict() == {"name": "Book", "price": 30000}
```

mix-in은 작고 독립적이어야 합니다. 내부 상태나 초기화 순서에 많이 의존하면 상속 관계가 복잡해집니다.

### Takeaway

mix-in은 “작은 기능 조각”일 때 유용합니다. 상태를 많이 가진 mix-in은 composition으로 바꾸는 편이 안전합니다.

## Item 55. Prefer Public Attributes over Private Ones

### Central Question

Python에서 private attribute는 어떤 의미일까?

### My Explanation

Python은 강제적 은닉보다 신뢰와 convention을 중시합니다.

```python
class User:
    def __init__(self, name: str) -> None:
        self.name = name          # public
        self._normalized = name.casefold()  # internal convention
```

single underscore는 “내부 구현으로 봐 달라”는 convention입니다. double underscore는 name mangling을 일으키지만, 완전한 보안 장치가 아닙니다.

```python
class Example:
    def __init__(self) -> None:
        self.__value = 10

example = Example()
assert hasattr(example, "_Example__value")
```

### Takeaway

Python에서는 public attribute를 기본으로 두고, 내부 구현은 `_name` convention으로 표시합니다. 과도한 은닉은 Python답지 않을 수 있습니다.

## Item 56. Prefer dataclasses for Creating Immutable Objects

### Central Question

불변 객체는 언제 유용할까?

### My Explanation

`frozen=True` dataclass는 생성 후 attribute 변경을 막습니다.

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Coordinate:
    x: int
    y: int

point = Coordinate(1, 2)
assert point == Coordinate(1, 2)
```

불변 객체는 hash key로 사용하기 좋고, 값이 중간에 변하지 않는다는 신뢰를 줍니다.

```python
visited = {Coordinate(1, 2)}
assert Coordinate(1, 2) in visited
```

### Takeaway

값 객체(value object)는 가능하면 불변으로 만들면 이해와 테스트가 쉬워집니다.

## Item 57. Inherit from collections.abc Classes for Custom Container Types

### Central Question

내 class가 container처럼 보이려면 어떤 protocol을 만족해야 할까?

### My Explanation

Python의 container는 duck typing으로 동작하지만, `collections.abc`를 사용하면 어떤 protocol을 구현하는지 명확해집니다.

```python
from collections.abc import Sequence

class Words(Sequence):
    def __init__(self, text: str) -> None:
        self._words = text.split()

    def __getitem__(self, index):
        return self._words[index]

    def __len__(self) -> int:
        return len(self._words)

words = Words("Python is clear")
assert len(words) == 3
assert words[0] == "Python"
assert list(words) == ["Python", "is", "clear"]
```

`Sequence`는 필요한 method를 구현하면 나머지 container behavior를 자연스럽게 사용할 수 있게 합니다.

### Takeaway

custom container를 만들 때는 `collections.abc`를 통해 어떤 protocol을 제공하는지 명시합니다.
