## Chapter Thesis

문자열과 슬라이싱은 Python 입문 문법처럼 보이지만, 실제로는 Python의 데이터 모델을 이해하는 핵심 입구입니다.

- `str`은 텍스트입니다.
- `bytes`는 인코딩된 이진 데이터입니다.
- slicing은 시퀀스 일부를 표현하는 문법입니다.
- formatting은 값을 문자열 세계로 번역하는 방식입니다.
- unpacking은 시퀀스의 구조를 코드로 드러내는 방식입니다.

이 장의 목표는 “문자열 메서드를 많이 아는 것”이 아니라, **텍스트/바이트/시퀀스의 경계를 정확히 구분하는 것**입니다.

## Item 10. Know the Differences Between bytes and str

### Central Question

Python에서 텍스트와 바이트는 왜 다른 타입이어야 할까?

### My Explanation

`str`은 사람이 읽는 텍스트를 표현합니다. `bytes`는 파일, 네트워크, 압축, 암호화, 프로토콜 등에서 오가는 이진 데이터를 표현합니다.

이 둘은 비슷해 보일 수 있지만 서로 다른 세계입니다.

```python
text = "파이썬"
data = text.encode("utf-8")

assert isinstance(text, str)
assert isinstance(data, bytes)
assert data.decode("utf-8") == text
```

핵심은 경계입니다.

- 프로그램 내부에서 의미 있는 텍스트를 다룰 때는 `str`
- 외부 시스템과 주고받는 인코딩된 데이터를 다룰 때는 `bytes`
- 두 세계를 오갈 때는 반드시 `encode`, `decode`를 명시

좋은 Python 코드는 입출력 경계에서만 인코딩을 처리하고, 내부 로직은 `str` 중심으로 유지합니다.

```python
def normalize_title(title: str) -> str:
    return title.strip().casefold()

raw = "  Python  ".encode("utf-8")
title = raw.decode("utf-8")

assert normalize_title(title) == "python"
```

반대로 내부 로직 곳곳에서 `bytes`와 `str`을 섞으면 코드가 쉽게 불안정해집니다.

```python
# TypeError: str과 bytes는 직접 결합할 수 없다.
# "prefix:" + b"data"
```

### Common Misunderstandings

- “영어만 쓰면 `str`과 `bytes`는 거의 같다.”  
  → 우연히 ASCII 범위에서 비슷해 보일 뿐, 타입의 의미는 다릅니다.

- “파일에서 읽은 것은 항상 문자열이다.”  
  → 텍스트 모드면 `str`, 바이너리 모드면 `bytes`입니다.

- “인코딩은 마지막에 대충 맞추면 된다.”  
  → 인코딩 경계는 설계 초기에 정해야 합니다.

### Takeaway

`str`은 의미이고, `bytes`는 표현입니다. Python 코드는 의미의 세계와 표현의 세계를 명확히 나눌 때 견고해집니다.

## Item 11. Prefer Interpolated F-Strings over C-Style Format Strings and str.format

### Central Question

문자열 formatting에서 왜 f-string이 기본 선택이 되었을까?

### My Explanation

f-string은 값과 표현 위치를 가깝게 둡니다. 그래서 문자열을 읽는 사람이 “어떤 값이 어디에 들어가는지” 바로 알 수 있습니다.

```python
name = "Ada"
score = 97.5

message = f"{name} scored {score:.1f}"
assert message == "Ada scored 97.5"
```

기존 `%` formatting이나 `str.format()`은 여전히 유효하지만, 값이 문자열에서 멀어질수록 읽기 비용이 커집니다.

```python
message = "%s scored %.1f" % (name, score)
message = "{} scored {:.1f}".format(name, score)
message = f"{name} scored {score:.1f}"
```

f-string의 debug specifier는 Python을 이해하는 데도 좋은 도구입니다.

```python
width = 10
height = 5
area = width * height

assert f"{area=}" == "area=50"
```

Python 3.14에는 template string literals, 즉 t-string도 있습니다. 하지만 일반적인 값 삽입과 사람이 읽는 문자열 생성에는 여전히 f-string이 기본 선택입니다. t-string은 문자열을 즉시 완성하기보다 interpolation 정보를 다룰 필요가 있는 더 특수한 상황에서 의미가 있습니다.

### Common Misunderstandings

- “f-string은 그냥 더 짧은 formatting이다.”  
  → 핵심은 짧음보다 지역성입니다. 값이 쓰이는 곳에 값이 보입니다.

- “f-string 안에는 아무 표현식이나 넣어도 좋다.”  
  → 복잡한 계산은 먼저 이름을 붙이는 편이 읽기 쉽습니다.

```python
ratio = success / total if total else 0
text = f"ratio={ratio:.2%}"
```

### Takeaway

f-string은 Python에서 가장 읽기 좋은 문자열 보간 방식입니다. 다만 f-string 안의 표현식이 복잡해지면 먼저 이름을 붙입니다.

## Item 12. Understand the Difference Between repr and str when Printing Objects

### Central Question

객체를 문자열로 보여줄 때, 누구를 위한 표현인가?

### My Explanation

`str(obj)`는 사용자 또는 일반 출력용 표현에 가깝고, `repr(obj)`는 개발자가 객체를 식별하고 재현하거나 디버깅하기 위한 표현에 가깝습니다.

```python
value = "Python\n"

assert str(value) == "Python\n"
assert repr(value) == "'Python\\n'"
```

`str`은 실제 줄바꿈을 포함한 텍스트처럼 보이고, `repr`은 문자열 안에 줄바꿈 문자가 있다는 사실을 드러냅니다.

```python
class Money:
    def __init__(self, amount: int, currency: str) -> None:
        self.amount = amount
        self.currency = currency

    def __str__(self) -> str:
        return f"{self.amount} {self.currency}"

    def __repr__(self) -> str:
        return f"Money(amount={self.amount!r}, currency={self.currency!r})"

price = Money(12000, "KRW")

assert str(price) == "12000 KRW"
assert repr(price) == "Money(amount=12000, currency='KRW')"
```

`__str__`은 자연스럽게 읽히게 하고, `__repr__`은 객체의 내부 상태를 드러냅니다.

f-string에서도 `!r`을 쓰면 `repr` 기반 출력이 됩니다.

```python
path = "C:\\temp\\data.txt"
assert f"{path!r}" == repr(path)
```

### Takeaway

`str`은 독자 친화적인 표현이고, `repr`은 개발자 친화적인 표현입니다. 객체를 설계할 때 둘의 독자를 다르게 상상해야 합니다.

## Item 13. Prefer Explicit String Concatenation over Implicit, Especially in Lists

### Central Question

문자열이 자동으로 붙는 기능은 언제 위험해질까?

### My Explanation

Python은 인접한 문자열 literal을 자동으로 이어 붙입니다.

```python
text = "hello" "world"
assert text == "helloworld"
```

이 기능은 긴 문자열을 여러 줄로 나눌 때 유용합니다.

```python
message = (
    "Python is expressive, "
    "but readability still matters."
)
```

하지만 list 안에서는 실수로 쉼표를 빠뜨렸을 때 조용히 다른 데이터가 됩니다.

```python
words = [
    "red",
    "green"
    "blue",
]

assert words == ["red", "greenblue"]
```

이런 종류의 오류는 실행은 되지만 의미가 틀립니다. 그래서 리스트나 튜플 안에서는 문자열 결합을 명시적으로 하는 편이 안전합니다.

여러 조각을 동적으로 붙일 때는 `join`이 의도를 더 잘 드러낼 때가 많습니다.

```python
parts = ["2026", "07", "02"]
date_text = "-".join(parts)

assert date_text == "2026-07-02"
```

### Takeaway

암묵적 문자열 결합은 제한적으로만 사용합니다. 특히 컬렉션 안에서는 쉼표 누락이 버그가 될 수 있으므로 명시성을 우선합니다.

## Item 14. Know How to Slice Sequences

### Central Question

슬라이스는 단순한 인덱스 문법인가, 아니면 시퀀스 일부를 표현하는 언어인가?

### My Explanation

슬라이스는 `sequence[start:stop]` 형태로 시퀀스의 일부를 표현합니다. 중요한 점은 `stop`이 포함되지 않는다는 것입니다.

```python
numbers = [0, 1, 2, 3, 4, 5]

assert numbers[1:4] == [1, 2, 3]
assert numbers[:3] == [0, 1, 2]
assert numbers[3:] == [3, 4, 5]
```

Python의 half-open interval은 길이를 계산하기 쉽습니다.

```python
start = 1
stop = 4
assert len(numbers[start:stop]) == stop - start
```

음수 인덱스는 끝에서부터 셉니다.

```python
letters = ["a", "b", "c", "d"]

assert letters[-1] == "d"
assert letters[-2:] == ["c", "d"]
```

list를 slicing하면 새 list가 만들어지지만 얕은 복사입니다.

```python
rows = [[1], [2]]
copy = rows[:]
copy[0].append(99)

assert rows == [[1, 99], [2]]
```

### Takeaway

slicing은 시퀀스의 구간을 표현하는 Python의 핵심 언어입니다. half-open 규칙과 얕은 복사를 함께 이해해야 합니다.

## Item 15. Avoid Striding and Slicing in a Single Expression

### Central Question

`sequence[start:stop:step]`은 언제 읽기 어려워질까?

### My Explanation

slice에는 step을 붙일 수 있습니다.

```python
numbers = [0, 1, 2, 3, 4, 5]

assert numbers[::2] == [0, 2, 4]
assert numbers[::-1] == [5, 4, 3, 2, 1, 0]
```

간단한 stride는 유용합니다. 그러나 start, stop, step이 모두 들어가고 특히 음수 step이 섞이면 해석 비용이 커집니다.

```python
numbers = list(range(10))
fragment = numbers[8:2:-2]
assert fragment == [8, 6, 4]
```

이럴 때는 단계로 나누는 편이 좋습니다.

```python
window = numbers[3:9]
fragment = window[::-2]

assert fragment == [8, 6, 4]
```

### Takeaway

stride와 slice를 동시에 복잡하게 쓰면 코드가 퍼즐이 됩니다. 복잡한 구간 추출은 단계로 나누어 의도를 드러냅니다.

## Item 16. Prefer Catch-All Unpacking over Slicing

### Central Question

시퀀스의 앞/중간/뒤를 나눌 때 slicing보다 unpacking이 더 나은 경우는 언제일까?

### My Explanation

catch-all unpacking은 `*rest`를 사용해 남은 값을 한 이름으로 받습니다.

```python
first, *middle, last = [1, 2, 3, 4, 5]

assert first == 1
assert middle == [2, 3, 4]
assert last == 5
```

slicing으로도 같은 일을 할 수 있지만, unpacking은 구조를 더 직접적으로 말합니다.

```python
command, *arguments = "commit -m initial".split()

assert command == "commit"
assert arguments == ["-m", "initial"]
```

길이가 다양한 입력을 다룰 때 slicing보다 자연스럽습니다. 단, 너무 복잡한 unpacking은 오히려 읽기 어렵습니다.

```python
# 구조가 단번에 이해되지 않는다.
a, *b, c, d = [1, 2, 3, 4, 5]
```

### Takeaway

catch-all unpacking은 시퀀스의 구조를 이름으로 드러냅니다. slicing이 위치 계산이라면, unpacking은 형태 선언입니다.
